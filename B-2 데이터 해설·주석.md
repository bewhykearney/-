# 1. 개요

데이터 해설·주석은 AI가 학습할 수 있도록 데이터에 의미 정보를 부여하는 활동이다.

AI는 원본 데이터만으로는 데이터의 의미를 이해하지 못한다. 예를 들어 결함 사진이 존재하더라도 어떤 결함인지, 어느 위치에 발생했는지, 심각도가 어느 수준인지에 대한 정보가 없으면 AI는 해당 데이터를 학습에 활용할 수 없다.

따라서 사람의 판단 기준을 구조화된 데이터 형태로 기록하여 AI가 활용 가능한 학습 데이터로 전환하는 과정이 필요하다.

AI-ready Data 관점에서 데이터 해설·주석은 단순한 라벨링 작업이 아니다. 현업의 판단 기준을 데이터 자산으로 전환하고, 반복적으로 재사용 가능한 학습 자산을 구축하는 활동이다.

## 1.1 AI-ready Data 체계 내 위치

데이터 해설·주석은 AI-ready Data 체계에서 전처리 이후 단계에 위치한다.

전처리가 데이터를 AI가 읽을 수 있는 형태로 변환하는 과정이라면, 데이터 해설·주석은 해당 데이터에 업무적 의미를 부여하여 AI가 학습할 수 있는 형태로 완성하는 과정이다.

```mermaid
flowchart LR

A[원본 데이터]
--> B[데이터 전처리]
--> C[데이터 해설·주석]
--> D[AI 학습 및 활용]
```

데이터 해설·주석은 AI 학습을 위한 정답 정보를 제공하는 역할을 수행하며, 평가 데이터 구축이나 품질 판정 자체를 담당하지는 않는다.

## 1.2 주요 참여 조직

데이터 해설·주석은 특정 조직만의 업무가 아니라 데이터 조직과 현업 조직이 공동으로 수행해야 하는 활동이다.

| 역할 | 주요 책임 |
|--------|--------|
| 전사 데이터 조직 | 표준 체계 수립, 도구 선정, 운영 정책 관리 |
| 현업 전문가 | 분류 체계 정의, 판단 기준 제공 |
| 라벨러 | 실제 주석 수행 |
| 검수자 | 품질 점검 및 합의 관리 |
| AI·데이터 조직 | 자동화, 버전 관리, 파이프라인 구축 |

특히 현업 전문가의 참여가 중요하다. 데이터 해설·주석은 결국 업무적 판단 기준을 데이터로 표현하는 과정이므로, 현업이 정의한 기준 없이는 품질을 확보하기 어렵다.

# 2. 왜 필요한가

AI 프로젝트에서 가장 흔하게 발생하는 문제는 모델의 성능 부족이 아니라 학습 데이터의 품질 부족이다.

많은 조직은 충분한 데이터를 보유하고 있다고 생각하지만 실제로는 AI가 학습할 수 있는 형태의 데이터가 준비되어 있지 않은 경우가 많다.

## 2.1 현업에서 발생하는 문제

첫 번째 문제는 정답 데이터의 부재이다.

예를 들어 동판 검사 공정에서 수만 장의 결함 이미지가 존재하더라도 결함 유형이 표시되어 있지 않다면 AI는 이를 학습에 활용할 수 없다.

두 번째 문제는 판단 기준의 불일치이다.

동일한 결함에 대해 작업자마다 다른 명칭을 사용하거나 다른 기준으로 분류하면 AI 역시 일관된 패턴을 학습하지 못한다.

세 번째 문제는 이력 관리의 부재이다.

누가 어떤 기준으로 주석을 작성했는지 추적할 수 없다면 데이터 품질을 지속적으로 개선하기 어렵다.

## 2.2 기대 효과

데이터 해설·주석 체계를 구축하면 세 가지 효과를 기대할 수 있다.

### 데이터 의미의 표준화

사람마다 다르게 해석하던 데이터를 공통 기준으로 관리할 수 있다.

### AI 학습 품질 향상

정확한 정답 데이터 확보를 통해 AI 성능을 안정적으로 향상시킬 수 있다.

### 데이터 자산화

한 번 구축한 주석 데이터는 이후 다양한 AI 과제에서 재사용할 수 있다.

예를 들어 동판 결함 이미지에 대한 표준 주석 데이터셋을 구축하면 외관 검사 자동화뿐 아니라 결함 원인 분석, 품질 예측, 설비 이상 탐지 과제에서도 활용할 수 있다.

## 2.3 AI-ready Data 관점의 의미

데이터 해설·주석은 단순히 AI 모델 하나를 위한 작업이 아니다.

장기적으로는 현업의 판단 기준을 조직의 데이터 자산으로 전환하고, 계열사 전반에서 재사용 가능한 형태로 축적하는 것이 목적이다.

따라서 주석 데이터는 프로젝트 종료와 함께 사라지는 산출물이 아니라 지속적으로 관리되는 데이터 자산으로 운영되어야 한다.

# 3. 무엇을 갖추나

데이터 해설·주석은 단순히 라벨을 부여하는 작업이 아니다.

AI가 활용할 수 있는 데이터를 지속적으로 구축하고 재사용하기 위해서는 대상 데이터 선정, 분류 체계 설계, 품질 검증, 운영 체계까지 포함하는 전체 프로세스가 필요하다.

또한 데이터 해설·주석은 프로젝트 단위 산출물이 아니라 향후 다양한 AI 과제에서 반복적으로 활용될 수 있는 데이터 자산이라는 관점에서 접근해야 한다.

본 장에서는 데이터 해설·주석 체계를 구성하는 핵심 요소와 운영 구조를 설명한다.

## 3.1 정본 모델

본 가이드는 아래 정본 모델을 기준으로 작성한다.

```text
대상 데이터 선정
→ Taxonomy 설계
→ Guideline 작성
→ Pilot Annotation
→ IAA 측정
→ 본 Annotation
→ QA
→ 자동화 및 자산화
```

데이터 해설·주석은 단순히 라벨을 붙이는 행위가 아니라 위 8개 단계를 통해 품질을 확보하고 재사용 가능한 데이터 자산을 구축하는 과정이다.

대상 데이터를 선정한 이후 분류 체계를 정의하고, 이를 기준으로 가이드라인을 구축한다.

이후 시범 주석 작업을 수행하여 기준의 적절성을 검증하고, 작업자 간 일관성을 확보한 뒤 본 주석 작업을 수행한다.

최종적으로는 품질 검증을 거쳐 재사용 가능한 데이터 자산으로 등록하고 운영한다.

본 정본 모델은 5장의 사례 시나리오, 6장의 실행 방법론, 8장의 Roadmap에서도 동일하게 사용한다.

## 3.2 Taxonomy와 라벨 정의 체계

Taxonomy는 데이터를 어떤 기준으로 분류할 것인지 정의하는 체계이다.

주석 품질은 결국 Taxonomy 품질에 의해 결정된다.

동일한 데이터를 서로 다른 기준으로 분류하면 AI는 일관된 패턴을 학습할 수 없으며, 작업자마다 다른 결과가 발생하게 된다.

따라서 주석 체계를 구축할 때 가장 먼저 수행해야 하는 작업 중 하나가 Taxonomy 설계이다.

좋은 Taxonomy는 다음 세 가지 조건을 만족해야 한다.

| 기준 | 설명 |
|--------|--------|
| 상호배타성 | 하나의 데이터가 여러 범주에 중복 포함되지 않음 |
| 포괄성 | 실제 발생 가능한 사례를 모두 수용 가능 |
| 일관성 | 동일한 기준으로 분류 체계 구성 |

예를 들어 동판 외관 검사 데이터를 대상으로 주석 체계를 설계한다고 가정하면 아래와 같은 구조를 사용할 수 있다.

| 코드 | 라벨 | 정의 |
|--------|--------|--------|
| DEF-01 | 균열 | 표면이 선형으로 갈라진 결함 |
| DEF-02 | 눌림 | 외부 압력에 의해 함몰된 결함 |
| DEF-03 | 오염 | 이물질 부착 또는 오염 상태 |
| DEF-04 | 스크래치 | 표면 긁힘 흔적 |

Taxonomy는 단순 분류표가 아니다.

향후 AI 모델의 학습 기준이 되며, 작업자 교육, 품질 검수, 자동화 체계의 기준으로 사용된다.

따라서 실제 구축 시에는 현업 전문가와 데이터 조직이 공동으로 설계해야 한다.

## 3.3 주석 가이드라인

Taxonomy가 무엇을 분류할 것인지 정의하는 체계라면, 주석 가이드라인은 어떻게 분류할 것인지를 정의하는 체계이다.

동일한 Taxonomy를 사용하더라도 작업자마다 다른 판단을 내릴 수 있다.

예를 들어 어떤 작업자는 특정 이미지를 균열로 분류하고, 다른 작업자는 스크래치로 분류할 수 있다.

이러한 차이를 줄이기 위해서는 판단 기준을 명문화한 주석 가이드라인이 필요하다.

주석 가이드라인은 최소한 다음 요소를 포함해야 한다.

| 항목 | 설명 |
|--------|--------|
| 정의 | 해당 라벨의 의미 |
| 포함 사례 | 해당 라벨에 포함되는 경우 |
| 제외 사례 | 유사하지만 포함되지 않는 경우 |
| 경계 사례 | 판단이 어려운 사례 |
| 결정 규칙 | 애매한 상황의 처리 기준 |

예를 들어 균열과 눌림을 구분하는 경우 아래와 같은 규칙을 정의할 수 있다.

```text
IF 선형으로 갈라진 흔적이 존재
→ 균열

IF 함몰 흔적은 존재하나 갈라짐은 없음
→ 눌림

IF 판단이 불가능함
→ 미상 처리 후 SME 검토
```

주석 가이드라인의 목적은 작업자의 판단 능력을 평가하는 것이 아니라 동일한 데이터를 동일하게 해석하도록 만드는 것이다.

## 3.4 데이터 유형별 주석 방식

주석은 데이터 유형에 따라 적용 방식이 달라진다.

이미지, 텍스트, 영상, 음성은 서로 다른 형태의 정보를 포함하고 있으며, 따라서 주석 방식 역시 달라져야 한다.

| 데이터 유형 | 대표 주석 방식 | 제조업 활용 예 |
|--------|--------|--------|
| 텍스트 | 문서 분류, 개체 추출, 구간 태깅 | 고객 클레임 분석 |
| 이미지 | 분류, 위치 표시, 영역 표시 | 결함 이미지 분석 |
| 영상 | 객체 추적, 이벤트 구간 표시 | 조립 공정 분석 |
| 음성 | 이벤트 탐지, 음성 전사 | 설비 이상음 분석 |

제조업에서는 이미지 데이터가 가장 대표적인 주석 대상이다.

특히 외관 검사, 품질 검사, 설비 점검 분야에서는 이미지 기반 AI 과제가 지속적으로 증가하고 있다.

다만 모든 과제가 정밀한 픽셀 수준의 주석을 필요로 하는 것은 아니다.

과제 목적에 따라 단순 분류, 위치 표시, 영역 표시 중 적절한 방식을 선택해야 한다.

## 3.5 작업자 간 일관성 확보 체계

데이터 해설·주석 품질은 개별 작업자의 정확성보다 작업자 간 일관성에 의해 결정된다.

따라서 조직은 여러 작업자가 동일한 결과를 도출할 수 있는 체계를 구축해야 한다.

이를 위해 작업자 간 일치도(IAA)를 활용한다.

IAA는 동일한 데이터를 여러 작업자가 주석했을 때 얼마나 동일한 결과를 생성하는지를 측정하는 지표이다.

일치도가 낮다는 것은 작업자 문제가 아니라 Taxonomy 또는 가이드라인 문제일 가능성이 높다.

예를 들어 아래와 같은 결과가 발생했다고 가정한다.

| 작업자 | 결과 |
|--------|--------|
| A | 균열 |
| B | 스크래치 |
| C | 균열 |

이 경우 B가 틀렸다고 판단하는 것이 아니라, 왜 다른 결과가 발생했는지를 분석해야 한다.

분석 결과에 따라 Taxonomy를 수정하거나 가이드라인을 보완할 수 있다.

결과적으로 IAA는 작업자를 평가하는 지표가 아니라 주석 체계를 개선하기 위한 품질 관리 지표이다.

## 3.6 데이터셋 버전 관리

주석 데이터는 시간이 지남에 따라 지속적으로 변경된다.

새로운 결함 유형이 추가될 수 있고, 기존 분류 체계가 변경될 수 있으며, 품질 검토 결과 일부 데이터가 재작업될 수도 있다.

따라서 주석 데이터는 일반 문서와 동일하게 버전 관리되어야 한다.

최소한 다음 정보는 관리해야 한다.

| 항목 | 설명 |
|--------|--------|
| 데이터셋 버전 | 버전 번호 |
| 변경 사유 | 변경 이유 |
| 작업자 | 수정 수행자 |
| 검수자 | 승인자 |
| 변경 일시 | 변경 시점 |
| 활용 이력 | 적용 모델 및 과제 |

버전 관리 체계는 단순한 이력 관리 목적만 갖지 않는다.

향후 데이터 자산화, 재사용, 품질 추적, 문제 분석의 기반이 된다.

따라서 데이터 해설·주석 체계를 구축할 때부터 버전 관리 방안을 함께 설계해야 한다.
