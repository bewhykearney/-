# 1. 개요

데이터 해설·주석은 AI가 학습할 수 있도록 데이터에 의미 정보를 부여하는 활동이다.

AI는 원본 데이터만으로는 데이터의 의미를 이해하지 못한다. 예를 들어 결함 사진이 존재하더라도 어떤 결함인지, 어느 위치에 발생했는지, 심각도가 어느 수준인지에 대한 정보가 없으면 AI는 해당 데이터를 학습에 활용할 수 없다.

따라서 사람의 판단 기준을 구조화된 데이터 형태로 기록하여 AI가 활용 가능한 학습 데이터로 전환하는 과정이 필요하다.

AI-ready Data 관점에서 데이터 해설·주석은 단순한 라벨링 작업이 아니다. 현업의 판단 기준을 데이터 자산으로 전환하고, 반복적으로 재사용 가능한 학습 자산을 구축하는 활동이다.

## 1.1 AI-ready Data 체계 내 위치

데이터 해설·주석은 AI-ready Data 체계에서 전처리 이후 단계에 위치한다.

전처리가 데이터를 AI가 읽을 수 있는 형태로 변환하는 과정이라면, 데이터 해설·주석은 해당 데이터에 업무적 의미를 부여하여 AI가 학습할 수 있는 형태로 완성하는 과정이다.

```mermaid
flowchart LR

A[원본 데이터]
--> B[데이터 전처리]
--> C[데이터 해설·주석]
--> D[AI 학습 및 활용]
```

데이터 해설·주석은 AI 학습을 위한 정답 정보를 제공하는 역할을 수행하며, 평가 데이터 구축이나 품질 판정 자체를 담당하지는 않는다.

## 1.2 주요 참여 조직

데이터 해설·주석은 특정 조직만의 업무가 아니라 데이터 조직과 현업 조직이 공동으로 수행해야 하는 활동이다.

| 역할 | 주요 책임 |
|--------|--------|
| 전사 데이터 조직 | 표준 체계 수립, 도구 선정, 운영 정책 관리 |
| 현업 전문가 | 분류 체계 정의, 판단 기준 제공 |
| 라벨러 | 실제 주석 수행 |
| 검수자 | 품질 점검 및 합의 관리 |
| AI·데이터 조직 | 자동화, 버전 관리, 파이프라인 구축 |

특히 현업 전문가의 참여가 중요하다. 데이터 해설·주석은 결국 업무적 판단 기준을 데이터로 표현하는 과정이므로, 현업이 정의한 기준 없이는 품질을 확보하기 어렵다.

# 2. 왜 필요한가

AI 프로젝트에서 가장 흔하게 발생하는 문제는 모델의 성능 부족이 아니라 학습 데이터의 품질 부족이다.

많은 조직은 충분한 데이터를 보유하고 있다고 생각하지만 실제로는 AI가 학습할 수 있는 형태의 데이터가 준비되어 있지 않은 경우가 많다.

## 2.1 현업에서 발생하는 문제

첫 번째 문제는 정답 데이터의 부재이다.

예를 들어 동판 검사 공정에서 수만 장의 결함 이미지가 존재하더라도 결함 유형이 표시되어 있지 않다면 AI는 이를 학습에 활용할 수 없다.

두 번째 문제는 판단 기준의 불일치이다.

동일한 결함에 대해 작업자마다 다른 명칭을 사용하거나 다른 기준으로 분류하면 AI 역시 일관된 패턴을 학습하지 못한다.

세 번째 문제는 이력 관리의 부재이다.

누가 어떤 기준으로 주석을 작성했는지 추적할 수 없다면 데이터 품질을 지속적으로 개선하기 어렵다.

## 2.2 기대 효과

데이터 해설·주석 체계를 구축하면 세 가지 효과를 기대할 수 있다.

### 데이터 의미의 표준화

사람마다 다르게 해석하던 데이터를 공통 기준으로 관리할 수 있다.

### AI 학습 품질 향상

정확한 정답 데이터 확보를 통해 AI 성능을 안정적으로 향상시킬 수 있다.

### 데이터 자산화

한 번 구축한 주석 데이터는 이후 다양한 AI 과제에서 재사용할 수 있다.

예를 들어 동판 결함 이미지에 대한 표준 주석 데이터셋을 구축하면 외관 검사 자동화뿐 아니라 결함 원인 분석, 품질 예측, 설비 이상 탐지 과제에서도 활용할 수 있다.

## 2.3 AI-ready Data 관점의 의미

데이터 해설·주석은 단순히 AI 모델 하나를 위한 작업이 아니다.

장기적으로는 현업의 판단 기준을 조직의 데이터 자산으로 전환하고, 계열사 전반에서 재사용 가능한 형태로 축적하는 것이 목적이다.

따라서 주석 데이터는 프로젝트 종료와 함께 사라지는 산출물이 아니라 지속적으로 관리되는 데이터 자산으로 운영되어야 한다.

# 3. 무엇을 갖추나

데이터 해설·주석은 단순히 라벨을 부여하는 작업이 아니다.

AI가 활용할 수 있는 데이터를 지속적으로 구축하고 재사용하기 위해서는 대상 데이터 선정, 분류 체계 설계, 품질 검증, 운영 체계까지 포함하는 전체 프로세스가 필요하다.

또한 데이터 해설·주석은 프로젝트 단위 산출물이 아니라 향후 다양한 AI 과제에서 반복적으로 활용될 수 있는 데이터 자산이라는 관점에서 접근해야 한다.

본 장에서는 데이터 해설·주석 체계를 구성하는 핵심 요소와 운영 구조를 설명한다.

## 3.1 정본 모델

본 가이드는 아래 정본 모델을 기준으로 작성한다.

```text
대상 데이터 선정
→ Taxonomy 설계
→ Guideline 작성
→ Pilot Annotation
→ IAA 측정
→ 본 Annotation
→ QA
→ 자동화 및 자산화
```

데이터 해설·주석은 단순히 라벨을 붙이는 행위가 아니라 위 8개 단계를 통해 품질을 확보하고 재사용 가능한 데이터 자산을 구축하는 과정이다.

대상 데이터를 선정한 이후 분류 체계를 정의하고, 이를 기준으로 가이드라인을 구축한다.

이후 시범 주석 작업을 수행하여 기준의 적절성을 검증하고, 작업자 간 일관성을 확보한 뒤 본 주석 작업을 수행한다.

최종적으로는 품질 검증을 거쳐 재사용 가능한 데이터 자산으로 등록하고 운영한다.

본 정본 모델은 5장의 사례 시나리오, 6장의 실행 방법론, 8장의 Roadmap에서도 동일하게 사용한다.

## 3.2 Taxonomy와 라벨 정의 체계

Taxonomy는 데이터를 어떤 기준으로 분류할 것인지 정의하는 체계이다.

주석 품질은 결국 Taxonomy 품질에 의해 결정된다.

동일한 데이터를 서로 다른 기준으로 분류하면 AI는 일관된 패턴을 학습할 수 없으며, 작업자마다 다른 결과가 발생하게 된다.

따라서 주석 체계를 구축할 때 가장 먼저 수행해야 하는 작업 중 하나가 Taxonomy 설계이다.

좋은 Taxonomy는 다음 세 가지 조건을 만족해야 한다.

| 기준 | 설명 |
|--------|--------|
| 상호배타성 | 하나의 데이터가 여러 범주에 중복 포함되지 않음 |
| 포괄성 | 실제 발생 가능한 사례를 모두 수용 가능 |
| 일관성 | 동일한 기준으로 분류 체계 구성 |

예를 들어 동판 외관 검사 데이터를 대상으로 주석 체계를 설계한다고 가정하면 아래와 같은 구조를 사용할 수 있다.

| 코드 | 라벨 | 정의 |
|--------|--------|--------|
| DEF-01 | 균열 | 표면이 선형으로 갈라진 결함 |
| DEF-02 | 눌림 | 외부 압력에 의해 함몰된 결함 |
| DEF-03 | 오염 | 이물질 부착 또는 오염 상태 |
| DEF-04 | 스크래치 | 표면 긁힘 흔적 |

Taxonomy는 단순 분류표가 아니다.

향후 AI 모델의 학습 기준이 되며, 작업자 교육, 품질 검수, 자동화 체계의 기준으로 사용된다.

따라서 실제 구축 시에는 현업 전문가와 데이터 조직이 공동으로 설계해야 한다.

## 3.3 주석 가이드라인

Taxonomy가 무엇을 분류할 것인지 정의하는 체계라면, 주석 가이드라인은 어떻게 분류할 것인지를 정의하는 체계이다.

동일한 Taxonomy를 사용하더라도 작업자마다 다른 판단을 내릴 수 있다.

예를 들어 어떤 작업자는 특정 이미지를 균열로 분류하고, 다른 작업자는 스크래치로 분류할 수 있다.

이러한 차이를 줄이기 위해서는 판단 기준을 명문화한 주석 가이드라인이 필요하다.

주석 가이드라인은 최소한 다음 요소를 포함해야 한다.

| 항목 | 설명 |
|--------|--------|
| 정의 | 해당 라벨의 의미 |
| 포함 사례 | 해당 라벨에 포함되는 경우 |
| 제외 사례 | 유사하지만 포함되지 않는 경우 |
| 경계 사례 | 판단이 어려운 사례 |
| 결정 규칙 | 애매한 상황의 처리 기준 |

예를 들어 균열과 눌림을 구분하는 경우 아래와 같은 규칙을 정의할 수 있다.

```text
IF 선형으로 갈라진 흔적이 존재
→ 균열

IF 함몰 흔적은 존재하나 갈라짐은 없음
→ 눌림

IF 판단이 불가능함
→ 미상 처리 후 SME 검토
```

주석 가이드라인의 목적은 작업자의 판단 능력을 평가하는 것이 아니라 동일한 데이터를 동일하게 해석하도록 만드는 것이다.

## 3.4 데이터 유형별 주석 방식

주석은 데이터 유형에 따라 적용 방식이 달라진다.

이미지, 텍스트, 영상, 음성은 서로 다른 형태의 정보를 포함하고 있으며, 따라서 주석 방식 역시 달라져야 한다.

| 데이터 유형 | 대표 주석 방식 | 제조업 활용 예 |
|--------|--------|--------|
| 텍스트 | 문서 분류, 개체 추출, 구간 태깅 | 고객 클레임 분석 |
| 이미지 | 분류, 위치 표시, 영역 표시 | 결함 이미지 분석 |
| 영상 | 객체 추적, 이벤트 구간 표시 | 조립 공정 분석 |
| 음성 | 이벤트 탐지, 음성 전사 | 설비 이상음 분석 |

제조업에서는 이미지 데이터가 가장 대표적인 주석 대상이다.

특히 외관 검사, 품질 검사, 설비 점검 분야에서는 이미지 기반 AI 과제가 지속적으로 증가하고 있다.

다만 모든 과제가 정밀한 픽셀 수준의 주석을 필요로 하는 것은 아니다.

과제 목적에 따라 단순 분류, 위치 표시, 영역 표시 중 적절한 방식을 선택해야 한다.

## 3.5 작업자 간 일관성 확보 체계

데이터 해설·주석 품질은 개별 작업자의 정확성보다 작업자 간 일관성에 의해 결정된다.

따라서 조직은 여러 작업자가 동일한 결과를 도출할 수 있는 체계를 구축해야 한다.

이를 위해 작업자 간 일치도(IAA)를 활용한다.

IAA는 동일한 데이터를 여러 작업자가 주석했을 때 얼마나 동일한 결과를 생성하는지를 측정하는 지표이다.

일치도가 낮다는 것은 작업자 문제가 아니라 Taxonomy 또는 가이드라인 문제일 가능성이 높다.

예를 들어 아래와 같은 결과가 발생했다고 가정한다.

| 작업자 | 결과 |
|--------|--------|
| A | 균열 |
| B | 스크래치 |
| C | 균열 |

이 경우 B가 틀렸다고 판단하는 것이 아니라, 왜 다른 결과가 발생했는지를 분석해야 한다.

분석 결과에 따라 Taxonomy를 수정하거나 가이드라인을 보완할 수 있다.

결과적으로 IAA는 작업자를 평가하는 지표가 아니라 주석 체계를 개선하기 위한 품질 관리 지표이다.

## 3.6 데이터셋 버전 관리

주석 데이터는 시간이 지남에 따라 지속적으로 변경된다.

새로운 결함 유형이 추가될 수 있고, 기존 분류 체계가 변경될 수 있으며, 품질 검토 결과 일부 데이터가 재작업될 수도 있다.

따라서 주석 데이터는 일반 문서와 동일하게 버전 관리되어야 한다.

최소한 다음 정보는 관리해야 한다.

| 항목 | 설명 |
|--------|--------|
| 데이터셋 버전 | 버전 번호 |
| 변경 사유 | 변경 이유 |
| 작업자 | 수정 수행자 |
| 검수자 | 승인자 |
| 변경 일시 | 변경 시점 |
| 활용 이력 | 적용 모델 및 과제 |

버전 관리 체계는 단순한 이력 관리 목적만 갖지 않는다.

향후 데이터 자산화, 재사용, 품질 추적, 문제 분석의 기반이 된다.

따라서 데이터 해설·주석 체계를 구축할 때부터 버전 관리 방안을 함께 설계해야 한다.

# 4. 어디부터 하나

데이터 해설·주석은 모든 데이터에 적용할 필요가 없다.

주석은 비용과 시간이 많이 소요되는 작업이므로 AI 활용 가치가 높고 사람의 판단이 필요한 데이터부터 우선 적용해야 한다.

따라서 데이터 해설·주석 체계를 구축할 때 가장 먼저 수행해야 하는 작업은 주석 대상 선정이다.

## 4.1 주석 대상 선정 기준

주석 대상은 다음 네 가지 기준을 활용하여 선정한다.

| 기준 | 설명 |
|--------|--------|
| 사람의 판단 필요성 | 사람의 해석이 있어야 의미를 이해할 수 있는가 |
| AI 활용 가능성 | AI 과제에 활용될 가능성이 높은가 |
| 데이터 확보 수준 | 충분한 데이터가 존재하는가 |
| 품질 영향도 | 업무 결과에 직접적인 영향을 미치는가 |

이 네 가지 기준을 모두 만족하는 데이터가 우선 적용 대상이 된다.

예를 들어 동판 외관 검사 이미지는 사람의 판단이 필요하고, AI 활용 가능성이 높으며, 품질에 직접적인 영향을 미치기 때문에 대표적인 주석 대상이다.

반면 생산 실적 데이터처럼 이미 코드값으로 관리되는 데이터는 별도 주석 없이 활용 가능한 경우가 많다.

## 4.2 우선 적용 대상

제조업 환경에서는 다음과 같은 데이터가 대표적인 주석 대상이 된다.

| 데이터 유형 | 활용 가치 | 우선순위 |
|--------|--------|--------|
| 결함 이미지 | 매우 높음 | 높음 |
| 고객 클레임 | 높음 | 높음 |
| 설비 이상 이력 | 높음 | 중간 |
| 시험 결과 보고서 | 중간 | 중간 |
| 일반 운영 데이터 | 낮음 | 낮음 |

특히 결함 이미지와 고객 클레임은 현업 판단 기준이 포함되어 있으며 AI 활용 효과도 크기 때문에 대부분의 제조 기업이 우선적으로 적용하는 영역이다.

## 4.3 적용하지 않아도 되는 경우

모든 데이터를 주석 대상으로 선정할 필요는 없다.

다음과 같은 경우에는 별도의 주석 체계 없이 활용 가능하다.

| 유형 | 이유 |
|--------|--------|
| 코드값 기반 데이터 | 이미 의미가 구조화되어 있음 |
| 단순 수치 데이터 | 별도 해석이 필요하지 않음 |
| 활용 계획이 없는 데이터 | 투자 대비 효과 부족 |
| 데이터 확보가 어려운 영역 | 운영 비용 과다 |

따라서 데이터 해설·주석은 "가능한 모든 데이터"가 아니라 "AI 활용 가치가 높은 데이터"에 집중해야 한다.

# 5. 예시 시나리오

본 장에서는 두산전자 동판 외관 검사 데이터를 활용하여 데이터 해설·주석 체계를 구축하는 사례를 설명한다.

이 사례는 3장에서 정의한 정본 모델을 실제 제조업 환경에 적용한 End-to-End 시나리오이다.

```text
대상 데이터 선정
→ Taxonomy 설계
→ Guideline 작성
→ Pilot Annotation
→ IAA 측정
→ 본 Annotation
→ QA
→ 자동화 및 자산화
```

## 5.1 대상 데이터 선정

두산전자에서는 동판 생산 과정에서 외관 검사 이미지를 지속적으로 생성하고 있다.

그러나 해당 이미지는 결함 유형이 표준화되어 있지 않으며 검사자마다 서로 다른 기준으로 기록되고 있다.

이에 따라 AI 기반 외관 검사 자동화를 위해 결함 이미지 데이터를 주석 대상으로 선정한다.

주요 대상은 다음과 같다.

- 균열
- 눌림
- 오염
- 스크래치

선정 기준은 다음과 같다.

| 항목 | 평가 |
|--------|--------|
| 사람 판단 필요성 | 높음 |
| AI 활용 가능성 | 높음 |
| 데이터 확보 수준 | 높음 |
| 품질 영향도 | 높음 |

## 5.2 Taxonomy 설계

주석 대상이 확정되면 결함 유형을 표준화한다.

```text
결함
├─ 균열
├─ 눌림
├─ 오염
└─ 스크래치
```

각 결함 유형에 대한 정의, 포함 사례, 제외 사례를 함께 정의한다.

예를 들어 균열은 선형으로 갈라진 흔적을 의미하며, 단순 표면 긁힘은 스크래치로 분류한다.

Taxonomy는 이후 모든 주석 작업의 기준이 된다.

## 5.3 Guideline 작성

Taxonomy가 정의되면 작업자들이 동일한 기준으로 주석할 수 있도록 가이드라인을 작성한다.

예를 들어 아래와 같은 규칙을 정의한다.

```text
IF 선형 갈라짐 존재
→ 균열

IF 표면 긁힘 흔적 존재
→ 스크래치

IF 판단 불가
→ 미상 처리 후 SME 검토
```

또한 실제 이미지 사례를 수집하여 포함 사례와 제외 사례를 정리한다.

이를 통해 작업자 간 해석 차이를 최소화한다.

## 5.4 Pilot Annotation

가이드라인이 작성되면 시범 주석 작업을 수행한다.

라벨러 3인이 동일한 이미지 500장을 독립적으로 주석한다.

Pilot 단계의 목적은 작업자를 평가하는 것이 아니라 가이드라인의 품질을 검증하는 것이다.

주요 산출물은 다음과 같다.

- 시범 주석 데이터셋
- 불일치 사례 목록
- 가이드라인 개선 항목

## 5.5 IAA 측정

Pilot 결과를 기반으로 작업자 간 일치도를 측정한다.

예를 들어 아래와 같은 결과가 나왔다고 가정한다.

| 항목 | 결과 |
|--------|--------|
| 균열 | 0.84 |
| 눌림 | 0.81 |
| 오염 | 0.88 |
| 스크래치 | 0.61 |

분석 결과 스크래치와 균열 구분 기준이 모호한 것으로 확인된다.

이에 따라 가이드라인을 수정하고 추가 사례를 보완한다.

이후 재검증을 수행하여 목표 수준을 달성한다.

## 5.6 본 Annotation

기준 검증이 완료되면 전체 데이터를 대상으로 본 주석 작업을 수행한다.

운영 효율성을 높이기 위해 AI가 1차 주석을 수행하고 사람은 검수 중심으로 참여한다.

```text
AI 1차 주석
→ 작업자 검토
→ SME 확인
→ 최종 승인
```

이를 통해 대규모 데이터셋을 효율적으로 구축할 수 있다.

## 5.7 QA

본 주석 작업이 완료되면 품질 검증을 수행한다.

주요 검증 항목은 다음과 같다.

| 항목 | 목적 |
|--------|--------|
| 라벨 정확도 | 잘못된 주석 제거 |
| 일관성 | 작업자 간 차이 확인 |
| 누락 여부 | 미주석 데이터 탐지 |
| 버전 검증 | 변경 이력 관리 |

QA를 통과한 데이터만 학습 데이터로 활용한다.

## 5.8 자동화 및 자산화

최종 데이터셋은 단순 프로젝트 산출물로 관리하지 않는다.

주석 데이터셋을 Registry에 등록하여 재사용 가능한 데이터 자산으로 운영한다.

| 관리 항목 | 예시 |
|--------|--------|
| Dataset ID | DS-ELEC-001 |
| 버전 | v1.0 |
| 활용 과제 | 외관 검사 AI |
| 생성 일자 | 2026-06 |
| 책임 조직 | 품질혁신팀 |

이후 해당 데이터셋은 외관 검사 자동화뿐 아니라 결함 원인 분석, 품질 예측, 이상 탐지 과제에서도 재사용할 수 있다.

결과적으로 데이터 해설·주석은 단순 라벨링 작업이 아니라 조직의 판단 기준을 데이터 자산으로 축적하는 활동이 된다.

# 6. 어떻게 준비·운영하나

본 장은 3장에서 정의한 정본 모델을 실제 운영 절차로 확장한 실행 매뉴얼이다.

```text
대상 데이터 선정
→ Taxonomy 설계
→ Guideline 작성
→ Pilot Annotation
→ IAA 측정
→ 본 Annotation
→ QA
→ 자동화 및 자산화
```

각 단계는 독립적으로 수행되지 않는다.

앞 단계의 산출물이 다음 단계의 입력값이 되며, 특정 단계에서 품질 문제가 발견될 경우 이전 단계로 돌아가 수정할 수 있다.

---

## 6.1 대상 데이터 선정

대상 데이터 선정 단계는 데이터 해설·주석 활동의 범위를 결정하는 단계이다.

주석 비용은 높고 유지관리 부담도 크기 때문에 AI 활용 가치가 높은 데이터부터 우선 선정해야 한다.

### 이 단계의 목적

| 목적 | 설명 |
|--------|--------|
| 우선순위 결정 | 주석 대상 데이터 확정 |
| 투자 효율 확보 | 효과가 높은 영역부터 적용 |
| 범위 관리 | 불필요한 주석 작업 방지 |

### 데이터팀이 수행해야 할 작업

| 순서 | 수행 작업 | 설명 |
|--------|--------|--------|
| 1 | 후보 데이터 조사 | 대상 데이터 목록 작성 |
| 2 | 활용 가능성 평가 | AI 활용 가능성 검토 |
| 3 | 우선순위 선정 | 적용 대상 확정 |

대상 데이터는 사람의 판단이 필요한 데이터인지 여부를 중심으로 검토해야 한다.

### 제조업 예시

| 항목 | 예시 |
|--------|--------|
| 데이터 | 동판 외관 검사 이미지 |
| 활용 목적 | 결함 자동 분류 |
| 선정 결과 | 우선 적용 대상 |

### 주요 산출물

- 대상 데이터 목록
- 우선순위 평가표
- 적용 범위 정의서

### 완료 기준

| 기준 | 설명 |
|--------|--------|
| 적용 대상 확정 | 대상 데이터 선정 완료 |
| 범위 정의 | 적용 범위 문서화 완료 |

---

## 6.2 Taxonomy 설계

Taxonomy 설계 단계에서는 데이터를 어떤 기준으로 분류할 것인지 정의한다.

이 단계의 품질은 이후 전체 주석 품질에 직접적인 영향을 미친다.

### 이 단계의 목적

| 목적 | 설명 |
|--------|--------|
| 분류 기준 정의 | 표준 라벨 체계 수립 |
| 일관성 확보 | 동일 데이터 동일 판단 유도 |

### 데이터팀이 수행해야 할 작업

| 순서 | 수행 작업 | 설명 |
|--------|--------|--------|
| 1 | 현업 인터뷰 | 판단 기준 수집 |
| 2 | 라벨 후보 정의 | 분류 체계 초안 작성 |
| 3 | SME 검토 | 기준 확정 |

### 제조업 예시

| 항목 | 예시 |
|--------|--------|
| 상위 분류 | 결함 |
| 하위 분류 | 균열, 눌림, 오염, 스크래치 |

### 주요 산출물

- Taxonomy 정의서
- 라벨 목록
- 코드 체계

### 완료 기준

| 기준 | 설명 |
|--------|--------|
| Taxonomy 승인 | SME 승인 완료 |
| 코드 정의 | 라벨 코드 체계 정의 완료 |

---

## 6.3 Guideline 작성

Guideline은 작업자들이 동일한 기준으로 주석할 수 있도록 만드는 핵심 문서이다.

### 이 단계의 목적

| 목적 | 설명 |
|--------|--------|
| 판단 기준 표준화 | 작업자 편차 최소화 |
| 품질 확보 | 재작업 감소 |

### 데이터팀이 수행해야 할 작업

| 순서 | 수행 작업 | 설명 |
|--------|--------|--------|
| 1 | 정의 작성 | 라벨 정의 정리 |
| 2 | 사례 수집 | 포함·제외 사례 작성 |
| 3 | 경계 사례 정리 | 애매한 사례 정리 |

### 제조업 예시

| 항목 | 예시 |
|--------|--------|
| 포함 사례 | 선형 갈라짐 |
| 제외 사례 | 단순 긁힘 |
| 경계 사례 | 얕은 균열 흔적 |

### 주요 산출물

- Guideline 문서
- 사례집
- 경계 사례 목록

### 완료 기준

| 기준 | 설명 |
|--------|--------|
| 가이드라인 승인 | 현업 승인 완료 |
| 사례 확보 | 대표 사례 확보 완료 |

---

## 6.4 Pilot Annotation

Pilot은 본 주석 작업 이전에 기준을 검증하는 단계이다.

### 이 단계의 목적

| 목적 | 설명 |
|--------|--------|
| 기준 검증 | Taxonomy 적절성 확인 |
| 품질 검증 | 작업자 편차 확인 |

### 데이터팀이 수행해야 할 작업

| 순서 | 수행 작업 | 설명 |
|--------|--------|--------|
| 1 | 샘플 선정 | 시범 데이터 구성 |
| 2 | 독립 주석 수행 | 작업자별 수행 |
| 3 | 결과 분석 | 차이점 분석 |

### 제조업 예시

| 항목 | 예시 |
|--------|--------|
| 대상 데이터 | 결함 이미지 500장 |
| 참여 인원 | 라벨러 3인 |

### 주요 산출물

- Pilot 데이터셋
- 불일치 사례 목록

### 완료 기준

| 기준 | 설명 |
|--------|--------|
| Pilot 완료 | 시범 작업 종료 |
| 분석 완료 | 결과 분석 종료 |

# 7. 다른 주제와의 관계

데이터 해설·주석은 AI-ready Data 체계에서 학습용 의미 정보를 생성하는 역할을 담당한다.

그러나 데이터 준비 과정 전체를 담당하는 것은 아니다.

전처리, 품질 관리, 온톨로지, 평가 데이터 등 인접 주제와 명확한 역할 분담이 필요하다.

본 장에서는 데이터 해설·주석이 담당하는 영역과 다른 주제의 영역을 구분한다.

## 7.1 데이터 전처리와의 관계

데이터 전처리는 데이터를 AI가 읽을 수 있는 형태로 변환하는 활동이다.

반면 데이터 해설·주석은 전처리가 완료된 데이터에 의미 정보를 부여하는 활동이다.

예를 들어 PDF 검사 보고서를 텍스트로 변환하는 것은 전처리의 영역이다.

반면 변환된 텍스트에 원인 유형이나 조치 유형을 부여하는 것은 데이터 해설·주석의 영역이다.

| 구분 | 데이터 전처리 | 데이터 해설·주석 |
|--------|--------|--------|
| 목적 | 읽을 수 있는 형태 변환 | 의미 정보 부여 |
| 입력 | 원본 데이터 | 전처리 완료 데이터 |
| 산출물 | 구조화 데이터 | 학습 데이터 |

따라서 데이터 해설·주석은 전처리 이후 단계에서 수행된다.

---

## 7.2 비즈니스 Glossary와의 관계

비즈니스 Glossary는 용어의 의미를 정의하는 체계이다.

반면 데이터 해설·주석은 데이터를 특정 용어로 분류하는 체계이다.

예를 들어 "균열"이라는 용어의 정의를 만드는 것은 Glossary의 역할이다.

반면 특정 이미지를 균열로 분류하는 것은 데이터 해설·주석의 역할이다.

| 구분 | 비즈니스 Glossary | 데이터 해설·주석 |
|--------|--------|--------|
| 관리 대상 | 용어 | 데이터 |
| 목적 | 의미 정의 | 의미 적용 |
| 산출물 | 표준 용어집 | 주석 데이터셋 |

Glossary는 데이터 해설·주석의 기준 정보를 제공한다.

---

## 7.3 온톨로지와의 관계

온톨로지는 개념 간 관계를 정의하는 체계이다.

반면 데이터 해설·주석은 개별 데이터를 특정 개념으로 분류하는 체계이다.

예를 들어 아래와 같은 관계는 온톨로지의 영역이다.

```text
균열
→ 원인 : 소재 이상

소재 이상
→ 조치 : 자재 교체
```

반면 실제 이미지 한 장을 균열로 분류하는 것은 데이터 해설·주석의 영역이다.

| 구분 | 온톨로지 | 데이터 해설·주석 |
|--------|--------|--------|
| 목적 | 개념 관계 정의 | 개별 데이터 분류 |
| 관리 단위 | 개념 | 데이터 |
| 활용 | 추론, 검색 | 학습 데이터 생성 |

---

## 7.4 데이터 품질 관리와의 관계

데이터 품질 관리는 데이터를 AI에 사용 가능한 수준인지 평가하는 체계이다.

반면 데이터 해설·주석은 데이터를 생성하는 체계이다.

| 구분 | 품질 관리 | 데이터 해설·주석 |
|--------|--------|--------|
| 목적 | 품질 판정 | 데이터 생성 |
| 결과 | 사용 가능 여부 | 주석 데이터셋 |
| 관점 | 품질 기준 | 의미 기준 |

주석 데이터 역시 품질 관리 대상이 된다.

따라서 데이터 해설·주석과 품질 관리는 상호 보완 관계에 있다.

---

## 7.5 AI 평가 데이터와의 관계

데이터 해설·주석과 AI 평가 데이터는 모두 정답 데이터를 다룬다는 점에서 혼동되기 쉽다.

그러나 목적이 다르다.

| 구분 | 데이터 해설·주석 | AI 평가 데이터 |
|--------|--------|--------|
| 목적 | 학습 | 평가 |
| 활용 시점 | 모델 학습 전 | 모델 학습 후 |
| 규모 | 대규모 | 소규모 고품질 |
| 역할 | 가르치는 정답 | 채점하는 정답 |

데이터 해설·주석은 AI에게 학습용 정답을 제공한다.

AI 평가 데이터는 학습이 끝난 모델을 평가하기 위한 기준 데이터이다.

따라서 동일한 데이터가 학습 데이터와 평가 데이터로 동시에 사용되어서는 안 된다.

---

## 7.6 데이터 Feedback Loop와의 관계

운영 과정에서 발견되는 오류는 데이터 Feedback Loop를 통해 다시 데이터 해설·주석 체계로 전달된다.

예를 들어 운영 중 특정 결함 유형의 오분류가 반복된다면 다음과 같은 흐름이 발생한다.

```text
AI 오분류 발생
→ Feedback 수집
→ 원인 분석
→ 주석 데이터 보완
→ 재학습
```

따라서 데이터 해설·주석은 데이터 Feedback Loop의 핵심 입력 요소 중 하나이다.

---

# 8. KPI 및 Roadmap

데이터 해설·주석은 프로젝트 단위 작업으로 끝나서는 안 된다.

지속적인 운영과 자동화, 자산화를 위해 KPI와 Roadmap을 함께 관리해야 한다.

## 8.1 KPI

### 품질 KPI

주석 품질은 데이터 해설·주석 체계의 가장 기본적인 성과 지표이다.

| KPI | 정의 | 측정 방식 |
|--------|--------|--------|
| 작업자 간 일치도 | 작업자 간 동일 판단 수준 | IAA |
| 라벨 오류율 | 잘못된 주석 비율 | 오류 건수 / 전체 건수 |
| QA 통과율 | 품질 검수 통과 비율 | 통과 건수 / 전체 건수 |

품질 KPI는 Taxonomy와 Guideline의 적절성을 검증하는 기준이 된다.

### 생산성 KPI

생산성 KPI는 주석 작업 효율을 측정한다.

| KPI | 정의 | 측정 방식 |
|--------|--------|--------|
| 처리량 | 시간당 주석 건수 | 총 건수 / 작업 시간 |
| 건당 비용 | 주석 1건당 비용 | 총 비용 / 총 건수 |

생산성 KPI는 자동화 효과를 측정하는 기준이 된다.

### 운영 KPI

운영 KPI는 주석 체계가 안정적으로 운영되는지를 측정한다.

| KPI | 정의 | 측정 방식 |
|--------|--------|--------|
| 재작업률 | 수정 필요 비율 | 재작업 건수 / 전체 건수 |
| 버전 변경 건수 | 기준 변경 빈도 | 월간 변경 건수 |

### 자동화 KPI

자동화 KPI는 AI 활용 수준을 측정한다.

| KPI | 정의 | 측정 방식 |
|--------|--------|--------|
| AI Pre-label 채택률 | AI 주석 승인 비율 | 승인 건수 / AI 생성 건수 |
| 자동 처리율 | 사람 개입 없는 처리 비율 | 자동 처리 건수 / 전체 건수 |

### 자산화 KPI

자산화 KPI는 데이터 재사용 수준을 측정한다.

| KPI | 정의 | 측정 방식 |
|--------|--------|--------|
| Registry 등록률 | 자산 등록 비율 | 등록 데이터셋 / 전체 데이터셋 |
| 데이터 재사용률 | 재사용 비율 | 재사용 데이터셋 / 전체 데이터셋 |
| Gold Dataset 활용률 | 표준 데이터 활용 비율 | 활용 과제 수 |

## Appendix A. 주요 용어

### 데이터 해설·주석

데이터에 의미 정보를 부여하여 AI가 학습 가능한 형태로 만드는 활동이다.

단순 데이터 입력 작업이 아니라 현업의 판단 기준을 데이터 자산으로 전환하는 과정이다.

---

### Taxonomy

데이터를 어떤 기준으로 분류할 것인지 정의하는 체계이다.

예를 들어 결함 데이터를 균열, 눌림, 오염, 스크래치로 구분하는 구조가 Taxonomy에 해당한다.

---

### Guideline

작업자들이 동일한 기준으로 데이터를 분류할 수 있도록 정의한 규칙 문서이다.

포함 사례, 제외 사례, 경계 사례, 결정 규칙 등을 포함한다.

---

### Annotation

데이터에 의미 정보를 부여하는 행위이다.

이미지의 경우 결함 위치를 표시하거나 결함 유형을 지정하는 작업이 대표적이다.

---

### SME

현업 전문가를 의미한다.

주석 기준 정의와 최종 검증 역할을 수행한다.

---

### IAA

작업자 간 일치도를 의미한다.

여러 작업자가 동일한 데이터를 얼마나 일관되게 분류하는지를 측정한다.

---

### Gold Dataset

전문가 검증이 완료된 고품질 데이터셋이다.

주석 품질 평가, AI 성능 평가, 작업자 교육 기준으로 활용된다.

---

### Registry

데이터 자산을 등록하고 관리하는 체계이다.

주석 데이터셋의 버전, 품질, 활용 이력 등을 관리한다.

## Appendix C. 작업자 합의 체계

작업자 간 불일치는 자연스럽게 발생한다.

중요한 것은 불일치를 줄이는 것이 아니라 동일한 기준으로 해결하는 것이다.

### 합의 방식

| 방식 | 설명 | 적용 상황 |
|--------|--------|--------|
| 다수결 | 가장 많은 선택 적용 | 일반 데이터 |
| SME 검토 | 전문가 최종 판단 | 고위험 데이터 |
| 합의 회의 | 공동 검토 | 반복 불일치 데이터 |
| 재정의 | 기준 자체 수정 | Taxonomy 문제 발생 |
