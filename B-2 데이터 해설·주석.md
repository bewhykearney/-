# Executive Summary

데이터 해설·주석은 AI가 데이터를 올바르게 이해하고 학습할 수 있도록 사람의 판단 기준과 의미 정보를 데이터에 부여하는 활동이다.

AI 모델의 성능은 알고리즘 자체보다 학습 데이터 품질의 영향을 크게 받는다. 동일한 데이터를 사용하더라도 라벨 기준이 일관되지 않거나 판단 근거가 명확하지 않으면 모델은 잘못된 패턴을 학습할 수 있으며, 이는 예측 정확도 저하와 운영 신뢰도 저하로 이어질 수 있다.

특히 제조 환경에서는 결함 이미지, 고객 클레임, 설비 점검 기록, 작업 일지와 같이 사람의 해석이 필요한 데이터가 다수 존재한다. 이러한 데이터는 원천 데이터만으로는 AI가 의미를 이해하기 어렵기 때문에, 데이터 해설·주석을 통해 의미 정보와 판단 기준을 명시적으로 부여해야 한다.

본 가이드는 데이터 해설·주석 체계를 구축하기 위해 필요한 주석 대상 선정, Taxonomy 설계, Annotation Guideline 작성, 품질 검증, 데이터셋 운영 및 자동화 방안을 정의한다.

이를 통해 계열사는 AI 학습 데이터 품질을 향상시키고, 주석 데이터셋을 재사용 가능한 자산으로 축적할 수 있다.

# 1. 개요

## 1.1 데이터 해설·주석이란

데이터 해설·주석(Data Annotation)은 AI가 데이터를 이해하고 학습할 수 있도록 데이터에 의미 정보를 부여하는 작업이다.

AI는 원본 데이터만으로는 데이터가 무엇을 의미하는지 스스로 이해하지 못한다. 예를 들어 결함 사진이 주어지더라도 AI는 해당 사진이 정상인지 불량인지, 어떤 유형의 결함인지, 어느 위치에 존재하는지 알 수 없다.

따라서 사람이 데이터에 의미 정보를 부여해야 하며, 이를 데이터 해설·주석이라고 한다.

예를 들어 결함 사진 한 장에 다음과 같은 정보를 추가할 수 있다.

| 구분 | 예시 |
|---|---|
| 결함 여부 | 불량 |
| 결함 유형 | 스크래치 |
| 결함 위치 | 외경 테두리 |
| 심각도 | 중 |
| 권장 조치 | 재가공 필요 |

이와 같이 원본 데이터에 사람이 해석한 의미 정보를 추가함으로써 AI가 학습 가능한 데이터로 전환된다.

---

## 1.2 데이터 해설·주석과 메타데이터의 차이

데이터 해설·주석은 메타데이터와 혼동되기 쉽지만 목적과 역할이 다르다.

메타데이터는 데이터를 설명하기 위한 정보이며, 데이터 해설·주석은 AI 학습과 판단을 위해 사람이 부여하는 의미 정보이다.

예를 들어 동일한 결함 사진에 대해 다음과 같이 구분할 수 있다.

| 구분 | 예시 |
|---|---|
| 메타데이터 | 촬영일시, 설비, 제품명, 해상도, 저장 위치 |
| 데이터 해설·주석 | 결함 유형, 결함 위치, 심각도, 원인, 권장 조치 |

즉 메타데이터는 "이 데이터가 무엇인가"를 설명하고, 데이터 해설·주석은 "이 데이터를 어떻게 해석해야 하는가"를 설명한다.

---

## 1.3 Understandable 영역 내 역할

Understandable 영역은 AI가 데이터를 올바르게 이해할 수 있도록 지원하는 영역이다.

세 가지 주제는 서로 다른 역할을 수행한다.

| 주제 | 역할 |
|---|---|
| 데이터 전처리 | AI가 읽을 수 있는 형태로 데이터를 정리하고 구조화 |
| 데이터 해설·주석 | AI 학습 및 판단을 위해 의미와 기준 부여 |
| 온톨로지 | 데이터와 업무 개념 간 관계를 구조화 |

데이터 전처리가 데이터의 형식을 정리하는 단계라면, 데이터 해설·주석은 데이터에 의미를 부여하는 단계이며, 온톨로지는 개념 간 관계를 체계화하는 단계이다.

---

## 1.4 AI-ready Data 체계 내 위치

데이터 해설·주석은 AI-ready Data 체계에서 AI가 데이터를 학습할 수 있도록 만드는 역할을 수행한다.

전처리를 통해 데이터가 AI가 읽을 수 있는 형태로 정리되면, 데이터 해설·주석을 통해 의미와 판단 기준이 부여된다.

```mermaid
flowchart LR

A[원본 데이터]
--> B[데이터 전처리]

B --> C[데이터 해설·주석]

C --> D[AI 학습]

D --> E[AI 활용]
```

데이터 전처리가 "읽을 수 있는 데이터"를 만드는 과정이라면, 데이터 해설·주석은 "학습할 수 있는 데이터"를 만드는 과정이다.

---

## 1.5 주요 대상 조직

데이터 해설·주석은 특정 조직만 수행하는 활동이 아니다.

주석 기준 정의, 실제 주석 수행, 품질 검수, AI 활용까지 다양한 조직이 함께 참여한다.

| 조직/역할 | 주요 역할 |
|---|---|
| 지주/전사 데이터 조직 | 주석 표준 및 운영 기준 수립 |
| 현업 전문가(SME) | 분류 기준 및 판단 기준 정의 |
| 데이터 라벨러 | 실제 라벨 및 주석 수행 |
| 검수자(QA) | 품질 검수 및 일치도 관리 |
| AI/데이터 조직 | AI 학습 및 자동화 체계 구축 |

현업 전문가의 역할이 특히 중요하다.

주석은 단순 입력 작업이 아니라 업무 지식을 데이터에 반영하는 과정이기 때문에, 도메인 전문가의 판단 기준이 반드시 반영되어야 한다.

---

# 2. 왜 필요한가 (Why)

AI 모델의 성능은 알고리즘보다 학습 데이터 품질의 영향을 크게 받는다.

동일한 모델을 사용하더라도 학습 데이터가 정확하고 일관되게 구축되어 있으면 높은 성능을 확보할 수 있지만, 데이터에 대한 해석 기준이 불명확하거나 라벨 품질이 낮으면 모델은 잘못된 패턴을 학습하게 된다.

특히 제조 환경에서는 결함 이미지, 고객 클레임, 작업 일지, 설비 점검 기록과 같이 사람의 해석이 필요한 데이터가 많다.

이러한 데이터는 원천 데이터만으로는 AI가 의미를 이해하기 어렵기 때문에, 사람의 판단 기준을 체계적으로 부여하는 데이터 해설·주석 체계가 필요하다.

---

## 2.1 데이터 해설·주석의 핵심 목적

데이터 해설·주석 체계는 단순히 라벨을 붙이는 작업이 아니라, AI가 데이터를 일관되게 이해할 수 있도록 만드는 기반 체계이다.

주요 목적은 다음과 같다.

### 데이터 의미 및 판단 기준 표준화

제품, 공정, 결함, 원인, 조치 등 핵심 업무 개념에 대한 해석 기준을 표준화한다.

이를 통해 동일한 데이터를 여러 사람이 보더라도 동일한 의미로 해석할 수 있도록 한다.

---

### 신뢰 가능한 AI 학습 환경 구축

표준화된 주석 기준과 검수 체계를 기반으로 AI 학습에 활용 가능한 고품질 데이터셋을 구축한다.

이를 통해 AI 모델이 잘못된 데이터를 학습하는 위험을 줄이고, 예측 결과의 신뢰도를 높일 수 있다.

---

### 주석 품질 및 이력 관리 체계 구축

주석 결과뿐 아니라 누가, 언제, 어떤 기준으로 주석을 생성했는지 추적 가능하도록 관리한다.

이를 통해 데이터 품질 문제 발생 시 원인을 추적하고 개선할 수 있다.

---

## 2.2 현업 Pain Point

많은 AI 과제는 모델 개발보다 학습 데이터 준비 단계에서 더 많은 시간과 비용이 소요된다.

대표적인 문제는 다음과 같다.

### 데이터 의미와 업무 맥락에 대한 표준 정의 부재

동일한 현상을 사람마다 다르게 해석하는 경우가 많다.

예를 들어 동일한 결함에 대해

- 스크래치
- 긁힘
- 표면 결함

등 서로 다른 표현을 사용할 수 있다.

이 경우 AI는 동일한 현상을 서로 다른 데이터로 학습하게 된다.

---

### 주석 기준 부재로 인한 품질 편차

주석 기준이 명확하지 않으면 동일한 데이터에 대해서도 작업자마다 다른 결과가 생성될 수 있다.

예를 들어 동일한 결함 사진을 보고

- 작업자 A는 크랙
- 작업자 B는 스크래치

로 분류할 수 있다.

이러한 불일치는 학습 데이터 품질을 저하시킨다.

---

### 주석 품질 및 변경 이력 관리 미흡

주석 데이터가 수정되더라도 변경 이력이 관리되지 않는 경우가 많다.

이 경우

- 어떤 기준으로 수정되었는지
- 어느 버전의 데이터가 모델 학습에 사용되었는지

확인하기 어렵다.

---

### AI 과제별 데이터 재작업 반복

과거 AI 과제에서 생성한 데이터셋이 존재함에도 재사용되지 못하는 경우가 많다.

그 결과

- 동일한 라벨링 반복
- 동일한 검수 반복
- 동일한 데이터 정제 반복

이 발생한다.

---

## 2.3 데이터 해설·주석 적용 효과

데이터 해설·주석 체계를 구축하면 다음과 같은 효과를 기대할 수 있다.

### 기계가 이해할 수 있는 데이터로 전환

원천 데이터에 의미 정보를 부여하여 AI가 학습 가능한 형태로 전환한다.

예를 들어 결함 사진은 단순 이미지 파일이지만,

주석을 통해

- 결함 유형
- 위치
- 심각도

가 추가되면 AI 학습 데이터가 된다.

---

### AI 학습 효율 향상

일관된 주석 기준을 적용하면 모델이 더 빠르게 학습하고 높은 정확도를 확보할 수 있다.

또한 데이터 품질 문제로 인한 재학습 비용을 줄일 수 있다.

---

### 데이터 재사용성 향상

표준화된 데이터셋은 여러 AI 과제에서 반복 활용할 수 있다.

예를 들어

품질 검사 데이터셋은

- 불량 분류 모델
- 결함 원인 분석
- 이상 탐지 모델

등 다양한 AI 과제에 활용 가능하다.

---

### AI 자산화 기반 구축

주석 데이터는 단순한 프로젝트 산출물이 아니라 재사용 가능한 AI 자산이다.

한 번 구축한 데이터셋은 지속적으로 축적하고 활용할 수 있어야 한다.

---

## 2.4 자회사 관점 기대 효과

자회사가 본 가이드를 적용하면 다음과 같은 효과를 기대할 수 있다.

| 구분 | 기대 효과 |
|---|---|
| 데이터 품질 | 주석 기준 표준화 |
| AI 성능 | 모델 정확도 및 신뢰도 향상 |
| 운영 효율 | 재작업 감소 |
| 비용 절감 | 데이터 구축 비용 절감 |
| 재사용성 | 데이터셋 재활용 가능 |
| 자산화 | AI 자산 지속 축적 |

---

## 2.5 데이터 해설·주석이 필요한 데이터

모든 데이터가 주석 대상은 아니다.

AI가 의미와 판단 기준을 직접 이해하기 어려운 데이터를 우선 대상으로 선정한다.

대표적인 대상은 다음과 같다.

| 데이터 유형 | 예시 |
|---|---|
| Text | 작업일지, 고객 클레임 |
| Image | 결함 이미지, 제품 사진 |
| Video | 작업 영상, 설비 영상 |
| Audio | 설비 이상음, 음성 기록 |

반대로 ERP 마스터 데이터처럼 의미가 이미 구조화되어 있는 데이터는 일반적으로 별도의 Annotation 대상이 아니다.

---

## 2.6 AI-ready Data 체계 내 역할 구분

데이터 해설·주석은 AI-ready Data 체계 내 다른 주제와 명확히 역할을 구분해야 한다.

| 주제 | 역할 |
|---|---|
| 데이터 전처리 | 데이터를 AI가 읽을 수 있도록 정리 |
| 데이터 해설·주석 | 의미와 판단 기준 부여 |
| 메타데이터 | 데이터 설명 정보 관리 |
| 온톨로지 | 개념 간 관계 정의 |
| 데이터 품질 관리 | AI 활용 가능 여부 평가 |

즉 데이터 해설·주석은 "데이터에 의미를 부여하는 역할"에 집중한다.

---

# 3. 무엇을 갖추나 (What)

데이터 해설·주석 체계는 AI가 데이터를 이해하고 학습할 수 있도록 의미 정보를 부여하는 체계이다.

본 가이드에서는 데이터 해설·주석 체계를 크게 다음 두 단계로 구분한다.

1. 데이터 라벨링(Labeling)
2. 데이터 주석(Annotation)

데이터 라벨링은 데이터 전체에 대한 분류 결과를 정의하는 활동이며, 데이터 주석은 해당 데이터에 대한 세부 속성과 맥락 정보를 추가하는 활동이다.

---

## 3.1 데이터 라벨링(Labeling)

데이터 라벨링은 데이터 전체에 대해 표준 분류 체계를 적용하는 활동이다.

AI 모델 학습에서는 라벨이 정답(Ground Truth)의 역할을 수행한다.

예를 들어 결함 이미지 한 장에 대해

- 정상
- 불량

을 부여하는 것이 가장 단순한 라벨링이다.

조금 더 세분화하면

- 스크래치
- 크랙
- 찍힘
- 오염

등으로 분류할 수 있다.

### 데이터 라벨링 예시

| 데이터 | 라벨 |
|---|---|
| 결함 이미지 | 스크래치 |
| 고객 클레임 | 배송 불만 |
| 설비 로그 | 이상 |
| 작업 일지 | 정비 작업 |

---

### 라벨 형태

라벨은 AI 과제 특성에 따라 다양한 형태로 구성될 수 있다.

#### 분류 라벨

정해진 범주 중 하나를 선택

예)

- 정상
- 경고
- 고장

---

#### 다중 클래스 라벨

여러 종류의 결함 또는 상태를 구분

예)

- 크랙
- 스크래치
- 찍힘
- 오염

---

#### 수치 라벨

연속적인 값을 표현

예)

- 심각도 1~5
- 위험도 0~100

---

#### 조치 라벨

권장 조치 정의

예)

- 재가공
- 폐기
- 재검사

---

## 3.2 데이터 주석(Annotation)

주석은 라벨만으로 표현하기 어려운 세부 정보와 맥락 정보를 추가하는 활동이다.

예를 들어

"스크래치"

라는 라벨만으로는

- 어디에 존재하는지
- 얼마나 심각한지
- 어떤 원인인지

를 알 수 없다.

이러한 정보를 구조화하여 추가하는 것이 Annotation이다.

---

### Annotation 예시

| 항목 | 예시 |
|---|---|
| 위치 | 외경 테두리 |
| 원인 | 소재 불량 |
| 심각도 | 중 |
| 부품 | 베어링 |
| 시간 | 2026-05-25 10:23 |

---

### 라벨과 주석의 차이

| 구분 | 라벨 | 주석 |
|---|---|---|
| 역할 | 분류 | 세부 설명 |
| 수준 | 데이터 전체 | 데이터 일부 |
| 예시 | 스크래치 | 위치=외경, 심각도=중 |

---

## 3.3 Annotation 유형

주석 대상 데이터는 크게 네 가지 유형으로 구분할 수 있다.

| 유형 | 예시 |
|---|---|
| Text | 작업일지, 고객 클레임 |
| Image | 결함 이미지 |
| Video | 생산 영상 |
| Audio | 설비 이상음 |

---

## 3.4 Text Annotation

Text Annotation은 텍스트 데이터에 의미 정보를 부여하는 작업이다.

### NER (Named Entity Recognition)

문서 내 주요 개체를 식별

예)

- 설비명
- 부품명
- 불량 유형

---

### Span Annotation

특정 구간에 의미 부여

예)

원인 분석 구간

조치 구간

---

### Relation Extraction

개체 간 관계 정의

예)

결함 → 원인

원인 → 조치

---

### Document Classification

문서 전체 분류

예)

- 품질 보고서
- 장애 보고서
- 작업 일지

---

## 3.5 Image Annotation

Image Annotation은 제조업에서 가장 많이 활용되는 주석 방식이다.

### Image Classification

이미지 전체에 라벨 부여

예)

정상

불량

---

### Bounding Box

객체 위치를 사각형으로 표시

적용 예

- 결함 탐지
- 부품 인식

---

### Polygon

객체 외곽선을 정밀하게 표시

적용 예

- 균열 영역 측정
- 부식 영역 측정

---

### Semantic Mask

모든 픽셀을 분류

적용 예

- 영역 분석
- 정밀 분할

---

### Keypoint

특정 위치를 좌표로 표시

적용 예

- 작업자 자세 분석
- 구조물 위치 분석

---

## 3.6 Video Annotation

Video Annotation은 시간 정보를 포함하는 주석 방식이다.

대표 유형

- Frame Classification
- Object Tracking
- Action Annotation
- Event Annotation
- Temporal Segment

---

## 3.7 Audio Annotation

Audio Annotation은 음성 및 소리 데이터에 의미를 부여하는 작업이다.

대표 유형

- Sound Event Annotation
- Speech-to-Text
- Speaker Annotation
- Intent Annotation
- Emotion Annotation

---

## 3.8 Taxonomy(분류 체계)

AI 학습을 위해서는 표준 분류 체계가 필요하다.

Taxonomy는 데이터를 중복 없이 일관되게 분류하기 위한 구조이다.

좋은 Taxonomy는 아래 세 가지 원칙을 만족해야 한다.

### 상호배타성

동일 데이터는 하나의 범주로 분류되어야 한다.

---

### 포괄성

모든 데이터가 적절한 범주를 가져야 한다.

---

### 일관성

동일 데이터는 항상 동일 기준으로 분류되어야 한다.

---

## 3.9 Taxonomy 구조 유형

### Flat

단일 레벨 구조

예)

정상

경고

고장

---

### Hierarchical

계층 구조

예)

설비 상태

→ 경고

→ 고장

→ 베어링 고장

---

### Multi-label

하나의 데이터에 여러 라벨 허용

예)

경고

+

온도 이상

+

베어링 이상

---

# 4. 어디부터 하나 — 주석 대상 선정

모든 데이터를 Annotation 대상으로 삼을 필요는 없다.

AI가 직접 의미와 판단 기준을 이해하기 어려운 데이터를 우선 대상으로 선정한다.

---

향후 AI 과제에 활용될 가능성이 높은가

---

### 재사용 가능한가

여러 과제에서 활용 가능한가

# 5. 예시 시나리오: 두산전자 결함 이미지 주석

본 장에서는 두산전자 외관 검사 데이터를 예시로 데이터 해설·주석 체계가 실제로 어떻게 적용되는지 설명한다.

데이터 해설·주석은 개념적으로는 단순해 보이지만, 실제 현장에서는 주석 기준 정의, 작업자 간 판단 차이, 품질 관리, AI 활용까지 고려해야 한다.

따라서 본 장에서는 두산전자 결함 이미지 사례를 통해 전체 흐름을 설명하고, 이후 6장에서 실제 구축 방법을 상세히 설명한다.

---

## 5.1 적용 배경

두산전자는 생산 과정에서 다양한 외관 결함 검사를 수행한다.

현재는 검사원이 결함 이미지를 육안으로 확인하여 정상 여부와 결함 유형을 판단하고 있다.

하지만 다음과 같은 문제가 존재한다.

### 동일 결함에 대한 표현 불일치

동일한 결함에 대해 작업자마다 다른 표현을 사용할 수 있다.

예)

| 작업자 | 표현 |
|---|---|
| A | 스크래치 |
| B | 긁힘 |
| C | 표면 손상 |

AI 입장에서는 동일한 결함임에도 서로 다른 데이터로 인식될 수 있다.

---

### 판단 기준 차이

동일한 이미지에 대해서도 작업자마다 다른 판정을 내릴 수 있다.

예)

| 작업자 | 판정 |
|---|---|
| A | 스크래치 |
| B | 크랙 |
| C | 스크래치 |

이 경우 학습 데이터 품질이 불안정해진다.

---

### 재사용 불가

과거에 구축한 결함 데이터가 존재하더라도

- 라벨 기준 불명확
- 작업자 정보 없음
- 버전 정보 없음

등의 이유로 재사용이 어렵다.

---

## 5.2 Annotation 적용 전

현재 상태에서는 결함 이미지가 존재하더라도 AI 학습에 직접 활용하기 어렵다.

```text
결함 이미지
↓
사람이 육안 판단
↓
결과 기록
```

구조로 운영되기 때문이다.

대표적인 문제는 다음과 같다.

### 데이터 의미 부재

이미지는 존재하지만 결함 의미가 정의되어 있지 않다.

---

### Ground Truth 부재

AI 학습을 위한 정답 데이터가 없다.

---

### 기준 불일치

검사원마다 다른 기준 사용

---

### 데이터 자산화 불가

프로젝트 종료 후 데이터가 활용되지 않는다.

---

## 5.3 Annotation 적용 후

표준 Annotation 체계를 적용하면 데이터에 의미 정보를 부여할 수 있다.

예시

| 항목 | 값 |
|---|---|
| 결함 여부 | 불량 |
| 결함 유형 | 스크래치 |
| 위치 | 외경 테두리 |
| 심각도 | 중 |
| 권장 조치 | 재가공 |

---

AI는 이러한 정보를 기반으로 학습을 수행한다.

```text
결함 이미지
↓
Annotation
↓
학습 데이터셋
↓
AI 모델
```

---

## 5.4 기대 효과

### 검사 기준 표준화

동일한 결함은 동일한 기준으로 분류

---

### AI 학습 가능

Ground Truth 확보

---

### 데이터 재사용 가능

다수 AI 과제 활용

---

### 검사 자동화

AI 기반 1차 판정 가능

---

### AI 자산 축적

주석 데이터셋을 지속적으로 재사용 가능

---

## 5.5 적용 이후 운영 모습

초기에는 사람이 직접 대부분의 주석을 수행한다.

```text
사람 80%
AI 20%
```

---

이후 데이터가 축적되면 AI가 1차 주석을 수행한다.

```text
AI Pre-label
↓
사람 검수
```

---

장기적으로는 사람이 어려운 사례만 검토한다.

```text
AI 90%
사람 10%
```

---

이를 통해 데이터 구축 비용을 절감하면서도 품질을 유지할 수 있다.

---

## 5.6 사례의 의미

두산전자 사례는 단순히 결함 이미지를 분류하는 사례가 아니다.

핵심은

- 판단 기준 표준화
- AI 학습 데이터 구축
- 데이터 재사용
- AI 자산 축적

체계를 만드는 것이다.

이후 6장에서는 이러한 체계를 실제로 구축하기 위한 방법론을 설명한다.

---

# 6. 어떻게 준비·운영하나 (How)

데이터 해설·주석 체계는 단순히 데이터를 라벨링하는 작업이 아니다.

표준 분류 체계를 설계하고, 판단 기준을 정의하며, 품질을 검증하고, AI 학습 데이터로 관리하기 위한 운영 체계 전체를 의미한다.

본 가이드에서는 데이터 해설·주석 체계를 아래 8단계로 구축하는 것을 권장한다.

```mermaid
flowchart LR

A[대상 데이터 선정]
--> B[Taxonomy 설계]

B --> C[Annotation Guideline 작성]

C --> D[Pilot Annotation]

D --> E[IAA 측정 및 보정]

E --> F[본 Annotation]

F --> G[QA 및 검수]

G --> H[데이터셋 버전 관리]
```

---

## 6.1 대상 데이터 선정

모든 데이터를 Annotation 대상으로 선정할 필요는 없다.

AI가 의미와 판단 기준을 직접 이해하기 어려운 데이터를 우선 대상으로 선정해야 한다.

---

### 선정 기준

#### 사람이 해석해야 하는가

예)

- 결함 이미지
- 고객 클레임
- 작업 일지

---

#### AI 활용 가치가 높은가

향후 AI 과제에서 반복 활용 가능한가

---

#### 데이터 확보가 가능한가

충분한 수량 확보가 가능한가

---

### 주요 산출물

- 대상 데이터 목록
- 샘플 데이터셋
- 우선순위 평가표

---

## 6.2 Taxonomy 설계

Taxonomy는 데이터를 분류하기 위한 표준 체계이다.

AI 학습 데이터의 품질은 Taxonomy 품질에 크게 영향을 받는다.

---

### 목적

- 동일 데이터를 동일 기준으로 분류
- 중복 Label 제거
- AI 학습 기준 제공

---

### 설계 원칙

#### 상호배타성

하나의 데이터는 하나의 기준으로 분류

예)

스크래치와 크랙이 동시에 같은 레벨에 존재하지 않도록 정의

---

#### 포괄성

모든 데이터를 분류 가능해야 함

예)

기타(Unknown) 클래스 포함

---

#### 일관성

동일 데이터는 항상 동일 Label 사용

---

### Before

| 표현 |
|---|
| 긁힘 |
| 스크래치 |
| 표면 손상 |

---

### After

| 표준 Label |
|---|
| 스크래치 |

---

### Taxonomy 유형

#### Flat Taxonomy

단일 레벨 구조

```text
정상
경고
고장
```

---

#### Hierarchical Taxonomy

계층 구조

```text
설비 상태
 ├ 정상
 ├ 경고
 │ ├ 온도 이상
 │ ├ 진동 이상
 └ 고장
```

---

#### Multi-label Taxonomy

하나의 데이터에 여러 Label 허용

```text
경고
+
온도 이상
+
베어링 이상
```

---

### 주요 산출물

- Taxonomy 구조도
- Label Definition
- Label Codebook

---

## 6.3 Annotation Guideline 작성

Taxonomy가 무엇을 분류할 것인지 정의하는 문서라면,

Annotation Guideline은 어떻게 판단할 것인지를 정의하는 문서이다.

동일한 데이터를 여러 사람이 일관되게 주석 처리하기 위해 필요하다.

---

### 왜 필요한가

주석 품질 문제는 작업자 역량보다 Guideline 품질에서 발생하는 경우가 많다.

대표적인 문제

- 기준이 모호함
- 예시 부족
- 경계 사례 미정의

---

### Guideline 필수 구성 요소

#### Objective

Annotation 목적

---

#### Definition

Label 정의

---

#### Scope

적용 범위

---

#### Rule

작업 규칙

---

#### Positive Example

정답 사례

---

#### Negative Example

오답 사례

---

#### Boundary Case

경계 사례

---

#### Decision Rule

판단 기준

---

### 좋은 Guideline의 조건

#### 결정 규칙 제공

예)

```text
IF 결함 길이 > 3mm
THEN 크랙

ELSE 스크래치
```

---

#### 예시와 반례 제공

정답 사례와 오답 사례를 함께 제공

---

#### 경계 사례 우선 정의

애매한 데이터를 먼저 정의

---

#### Feedback Loop 운영

실제 결과를 반영해 지속 개선

---

#### 버전 관리

Guideline 변경 이력 기록

---

### 주요 산출물

- Annotation Guideline
- 사례집
- Edge Case 목록

---

## 6.4 Pilot Annotation

Pilot Annotation은 본 작업 이전에 수행하는 시범 Annotation이다.

목적은 Annotation 체계가 실제로 동작하는지 검증하는 것이다.

---

### 수행 절차

#### 데이터 분할

| 구분 | 비율 |
|---|---|
| Training | 60~70% |
| Test | 20~30% |
| Hold-out | 10% |

---

#### Ground Truth 구축

전문가 검증 수행

권장 규모

100~300건

---

#### Pilot 수행

다수 작업자 참여

---

#### 결과 분석

불일치 사례 확인

---

### 성능 지표

#### Accuracy

정확도

---

#### Precision / Recall

정밀도 / 재현율

---

#### F1 Score

종합 지표

---

#### Cohen's Kappa

작업자 일치도

---

### 주요 산출물

- Pilot 결과 보고서
- 품질 이슈 목록
- Guideline 개선안

---

## 6.5 IAA 측정 및 Guideline 보정

IAA(Inter-Annotator Agreement)는

여러 작업자가 동일 데이터를 얼마나 일관되게 해석하는지 측정하는 지표이다.

---

### 왜 필요한가

동일 데이터를

작업자 A는 크랙

작업자 B는 스크래치

로 분류한다면

학습 데이터 품질은 낮아질 수밖에 없다.

---

### 대표 지표

| 지표 | 적용 대상 |
|---|---|
| Cohen's Kappa | 2인 |
| Fleiss' Kappa | 3인 이상 |
| Krippendorff's Alpha | 다수 작업자 |

---

### 권장 기준

| 값 | 수준 |
|---|---|
| 0.81~1.0 | 매우 높음 |
| 0.61~0.80 | 상당 |
| 0.41~0.60 | 보통 |
| 0.40 이하 | 재검토 |

---

실무 목표

```text
IAA ≥ 0.80
```

---

### IAA 미달 시 조치

- Taxonomy 수정
- Guideline 수정
- 예시 추가
- 작업자 교육
- Pilot 재수행

---

## 6.6 Labeler 합의 체계

IAA 측정 이후에도 불일치 데이터는 발생한다.

따라서 합의 체계를 정의해야 한다.

---

### 대표 방식

#### 다수결

가장 많은 Label 채택

---

#### 가중 다수결

전문성에 따라 가중치 적용

---

#### 순차 검토

1차 작업

↓

시니어 검토

---

#### 다층 QC

Peer Review

↓

Central Review

↓

Final Approval

---

### 운영 목적

- 품질 향상
- 편차 감소
- Ground Truth 확보

---
## 6.7 Gold Standard Dataset 구축

Gold Standard Dataset은 전문가가 검증한 정답 데이터셋으로, 데이터 해설·주석 체계 전체의 기준점 역할을 수행한다.

Annotation Guideline이 판단 기준을 문서로 정의한 것이라면, Gold Standard Dataset은 해당 기준을 실제 데이터에 적용한 모범 사례 집합이다.

---

### 왜 필요한가

주석 작업이 장기간 진행되면 작업자마다 기준이 달라질 수 있다.

이를 Annotation Drift라고 하며, 시간이 지날수록 데이터 품질이 저하되는 주요 원인 중 하나이다.

Gold Standard Dataset은 이러한 Drift를 방지하고 품질을 유지하기 위한 기준점 역할을 수행한다.

---

### 주요 활용 목적

#### 작업자 품질 평가

새로운 작업자가 동일한 기준으로 주석을 수행하는지 검증

---

#### AI 모델 검증

자동 Annotation 결과의 품질 검증

---

#### Guideline 정렬

주석 기준 변경 여부 판단

---

#### 교육 자료 활용

신규 작업자 교육

---

### 구축 원칙

#### 전문가 검증

도메인 전문가가 직접 검증

---

#### 다양한 사례 포함

정상 사례

대표 사례

희귀 사례

경계 사례 포함

---

#### 변경 이력 관리

버전 단위 관리

---

### 구축 절차

```text
대표 데이터 선정
→ 전문가 라벨링
→ 검토 및 합의
→ 최종 승인
→ Gold Dataset 등록
```

---

### 권장 규모

초기 구축

200~500건

---

운영 단계

500~1000건

---

## 6.8 본 Annotation 수행

Pilot과 품질 검증이 완료되면 본 Annotation을 수행한다.

---

### 수행 절차

데이터 수집
→ 작업 단위 생성
→ Annotation 수행

→ 데이터셋 등록

### 주요 산출물

- 학습 데이터셋

- 운영 로그

---

### 운영 관리 항목

#### 생산성

- 처리 건수
- 처리 시간
- 작업량

---

#### 품질

- 오류율
- 재작업률
- IAA

---

#### 예외 사항

- 신규 유형 발견
- Taxonomy 확장
- Guideline 수정

---

## 6.9 QA 및 검수

본 Annotation 완료 후 데이터 품질을 검증한다.

---

### 검수 목적

- 오류 제거
- 품질 확보
- 학습 데이터 신뢰성 확보

---

### 주요 검수 항목

#### Label 정확도

올바른 Label 적용 여부

---

#### Annotation 누락 여부

필수 항목 누락 여부

---

#### Guideline 준수 여부

정의된 기준 준수 여부

---

#### 형식 검증

데이터 포맷 확인

---

### 검수 결과 처리

| 결과 | 조치 |
|---|---|
| 승인 | 데이터셋 등록 |
| 수정 | 작업자 재수정 |
| 반려 | 재작업 |

---

## 6.10 AI 기반 Annotation 자동화

초기에는 사람이 직접 대부분의 Annotation을 수행한다.

그러나 데이터가 충분히 축적되면 AI를 활용하여 Annotation 생산성을 향상시킬 수 있다.

---

### 자동화 목적

- 비용 절감
- 처리 속도 향상
- 품질 일관성 확보

---

### 적용 원칙

사람을 제거하는 것이 아니라

사람이 해야 할 작업을 줄이는 것이 목적이다.

---

## 6.11 Pre-labeling

Pre-labeling은 AI가 먼저 Annotation 초안을 생성하는 방식이다.

작업자는 처음부터 데이터를 작성하는 대신 결과를 검토하고 수정한다.

---

### 효과

- 작업 시간 감소
- 생산성 향상
- 품질 일관성 향상

---

### 예시

```text
원본 이미지
→ AI 자동 Label
→ 사람 검토
→ 최종 승인
```

---

## 6.12 Human-in-the-Loop(HITL)

AI가 생성한 결과를 사람이 검증하는 구조이다.

---

### 운영 원칙

신뢰도(Confidence)에 따라 처리 방식을 다르게 적용한다.

| 신뢰도 | 처리 방식 |
|---|---|
| 0.90 이상 | 자동 승인 |
| 0.70~0.90 | Reviewer 검토 |
| 0.70 미만 | SME 검토 |

---

### 기대 효과

- 검수 비용 절감
- 품질 유지
- 전문가 집중 활용

---

## 6.13 Active Learning

Active Learning은 AI가 가장 어려워하는 데이터를 우선적으로 선택하여 사람에게 검토를 요청하는 방식이다.

---

### 목적

동일한 비용으로 더 높은 모델 성능 확보

---

### 대표 방식

#### Uncertainty Sampling

가장 확신이 낮은 데이터 선택

---

#### Diversity Sampling

다양한 데이터 우선 선택

---

#### Core-set Sampling

대표 데이터 우선 선택

---

### 기대 효과

- 라벨링 비용 절감
- 희귀 사례 확보
- 학습 효율 향상

---

## 6.14 Weak Supervision

Weak Supervision은 사람이 직접 모든 데이터를 Annotation하지 않고 규칙을 통해 대량의 Label을 생성하는 방식이다.

---

### 활용 예시

#### 규칙 기반

```text
온도 > 100℃
→ 고온 이상
```

---

#### 키워드 기반

```text
"스크래치"
포함
→ 스크래치 결함
```

---

### 기대 효과

- 대규모 데이터 처리
- Annotation 비용 절감
- 초기 데이터 확보

---

## 6.15 Annotation 데이터셋 버전 관리

Annotation 데이터는 지속적으로 변경된다.

따라서 버전 관리 체계를 구축해야 한다.

---

### 버전 관리 대상

- Taxonomy
- Guideline
- Annotation Dataset
- Gold Standard Dataset

---

### 버전 규칙

| 유형 | 설명 |
|---|---|
| Major | Taxonomy 변경 |
| Minor | Label 추가 |
| Patch | 오류 수정 |

---

### 예시

| 버전 | 변경 사항 |
|---|---|
| v1.0 | 초기 구축 |
| v1.1 | Label 오류 수정 |
| v1.2 | Guideline 보완 |
| v2.0 | Taxonomy 확장 |

---

### 버전 관리 목적

#### 재현성 확보

동일 조건으로 모델 재학습 가능

---

#### 추적성 확보

누가, 언제, 무엇을 변경했는지 확인

---

#### 안정성 확보

문제 발생 시 이전 버전 복구 가능

---

### 최종 운영 목표

데이터 해설·주석 체계는 단순 Annotation 프로젝트가 아니라 지속적으로 학습 데이터와 AI 자산을 축적하는 운영 체계로 발전해야 한다.

이를 통해

- 학습 데이터 재사용
- Annotation 비용 절감
- AI 성능 향상
- 자산화

를 달성할 수 있다.

---
# 7. 다른 주제와의 관계

데이터 해설·주석은 독립적으로 운영되는 체계가 아니다.

AI-ready Data 체계 내 다른 주제와 연계하여 운영해야 한다.

특히 데이터 전처리, 메타데이터, 비즈니스 Glossary, 온톨로지, 데이터 품질 관리와의 역할을 명확히 구분해야 한다.

---

## 7.1 데이터 전처리와의 관계

데이터 전처리는 데이터를 AI가 읽을 수 있는 형태로 정리하는 과정이다.

반면 데이터 해설·주석은 AI가 학습할 수 있도록 의미 정보를 부여하는 과정이다.

| 구분 | 역할 |
|---|---|
| 데이터 전처리 | 데이터 정제, 변환, 구조화 |
| 데이터 해설·주석 | 의미 및 판단 기준 부여 |

---

### 예시

원본 작업일지

```text
베어링 교체 완료
온도 상승 확인
```

---

전처리 결과

```json
{
  "설비":"A-101",
  "내용":"베어링 교체 완료"
}
```

---

주석 결과

```json
{
  "설비":"A-101",
  "작업유형":"예방정비",
  "이상유형":"온도상승"
}
```

---

## 7.2 메타데이터와의 관계

메타데이터는 데이터를 설명하기 위한 정보이다.

반면 데이터 해설·주석은 AI 학습과 판단을 위해 사람이 부여하는 의미 정보이다.

---

### 메타데이터

- 생성일시
- 저장 위치
- 담당 조직
- 단위
- 설명

---

### 데이터 해설·주석

- 결함 유형
- 원인
- 심각도
- 조치
- 상태

---

## 7.3 비즈니스 Glossary와의 관계

비즈니스 Glossary는 업무 용어의 표준 의미를 정의한다.

데이터 해설·주석은 해당 용어를 실제 데이터에 적용한다.

---

### 예시

Glossary

```text
스크래치
=
제품 표면의 선형 긁힘 결함
```

---

Annotation

```text
이미지 #1034
→ 스크래치
```

---

## 7.4 온톨로지와의 관계

온톨로지는 개념 간 관계를 정의한다.

데이터 해설·주석은 개별 데이터에 의미를 부여한다.

---

### Annotation

```text
결함 = 크랙
```

---

### Ontology

```text
크랙
→ 원인
→ 소재 불량

크랙
→ 조치
→ 재가공
```

---

## 7.5 데이터 품질 관리와의 관계

데이터 품질 관리는 데이터를 AI에 활용할 수 있는지 평가한다.

데이터 해설·주석은 AI 학습 데이터를 구축한다.

---

### 역할 구분

| 주제 | 역할 |
|---|---|
| 데이터 해설·주석 | 학습 데이터 생성 |
| 데이터 품질 관리 | 데이터 사용 가능 여부 평가 |

---

## 7.6 AI 평가 데이터와의 관계

데이터 해설·주석은 학습 데이터를 구축한다.

AI 평가 데이터는 모델을 평가하기 위한 정답 데이터셋을 구축한다.

---

### 역할 차이

| 구분 | 목적 |
|---|---|
| 데이터 해설·주석 | AI 학습 |
| AI 평가 데이터 | AI 평가 |

---

# 8. KPI 및 성과 관리

데이터 해설·주석 체계는 품질뿐 아니라 생산성, 자동화 수준까지 함께 관리해야 한다.

KPI는 다음 네 가지 관점으로 관리한다.

- 품질
- 생산성
- 운영
- 자동화

---

## 8.1 KPI 운영 원칙

단순히 Annotation 수량만 관리하는 것이 아니라

- 얼마나 정확한가
- 얼마나 일관적인가
- 얼마나 효율적인가
- 얼마나 자동화되었는가

를 함께 측정한다.

---

## 8.2 품질 KPI

### IAA (Inter-Annotator Agreement)

작업자 간 일관성 측정

---

정의

동일 데이터를 여러 작업자가 얼마나 동일하게 해석하는지 측정

---

활용 단계

```text
Pilot Annotation
IAA 측정
Guideline 보정
```

---

목표

```text
IAA ≥ 0.80
```

---

### 라벨 오류율 (LER)

정의

검수 과정에서 발견된 오류 비율

---

산식

```text
오류 라벨 수
÷
검수 라벨 수
```

---

활용 단계

```text
QA
품질 점검
```

---

## 8.3 생산성 KPI

### 처리량

정의

단위 시간당 Annotation 처리 건수

---

산식

```text
완료 건수
÷
작업 시간
```

---

### 건당 비용

정의

데이터 1건 구축 비용

---

산식

```text
총 비용
÷
완료 건수
```

---

활용 단계

```text
본 Annotation 수행
```

---

## 8.4 운영 KPI

### QA 통과율

정의

검수 통과 비율

---

### 재작업률

정의

수정 또는 재작업 비율

---

활용 단계

```text
QA 및 검수
```

---

## 8.5 자동화 KPI

### AI Pre-label 채택률

정의

AI가 생성한 Annotation이 수정 없이 승인된 비율

---

산식

```text
수정 없이 승인된 AI 라벨
÷
전체 AI 생성 라벨
```

---

활용 단계

```text
자동화 적용 이후
```

---

## 8.6 KPI Dashboard 예시

관리 항목

- IAA
- 오류율
- 처리량
- 재작업률
- AI Pre-label 채택률

---

## 8.7 자회사 적용 시 기대 효과

KPI를 지속적으로 운영하면 다음 효과를 기대할 수 있다.

- Annotation 품질 향상
- 재작업 감소
- 구축 비용 절감
- AI 학습 데이터 재사용 확대
- 자동화 수준 향상

# 9. Roadmap

## 9.1 추진 방향

데이터 해설·주석 체계는 단순 수작업 중심 체계에서 AI 기반 자동화 체계로 발전해야 한다.

```text
표준화
→ 구축
→ 운영
→ 자동화
→ AI 자산화
```

---

## Phase 1. Preparation

### 목표

신뢰 가능한 Annotation 기준 구축

---

### 주요 활동

- Taxonomy 정의
- Guideline 작성
- Gold Standard Dataset 구축
- Pilot Annotation
- IAA 기준선 확보

---

### 주요 산출물

- Taxonomy
- Annotation Guideline
- Gold Standard Dataset

---

### 전환 조건

```text
IAA ≥ 0.80
```

---

## Phase 2. AI Ready

### 목표

대규모 Annotation 체계 구축

---

### 주요 활동

- 본 Annotation 수행
- QA 체계 구축
- 버전 관리 체계 구축
- AI Pre-label 적용

---

### 주요 산출물

- Annotation Dataset
- QA Report
- Version 관리 체계

---

### 전환 조건

```text
AI Pre-label 채택률 ≥ 50%
```

---

## Phase 3. Automation

### 목표

자동화 기반 운영 체계 구축

---

### 주요 활동

- Active Learning
- Weak Supervision
- HITL 운영
- 자동 QA

---

### 주요 산출물

- 자동화 Pipeline
- AI Annotation 체계

---

### 전환 조건

```text
AI Pre-label 채택률 ≥ 80%
```

---

## Phase 4. AI Assetization

### 목표

주석 데이터를 재사용 가능한 AI 자산으로 운영

---

### 주요 활동

- Annotation Dataset 재사용
- Prompt 연계
- 평가 데이터 연계
- AI 자산 관리

---

### 주요 산출물

- AI 자산 Repository
- Annotation Asset Catalog

---

## 9.2 자회사 적용 가이드

자회사는 현재 수준에 따라 시작 단계를 결정한다.

| 현재 상태 | 권장 시작 Phase |
|---|---|
| 주석 체계 없음 | Phase 1 |
| 일부 AI 과제 수행 중 | Phase 2 |
| AI 운영 조직 보유 | Phase 3 |

---

## 9.3 최종 목표

데이터 해설·주석 체계의 최종 목표는 단순히 AI 학습 데이터를 만드는 것이 아니다.

사람의 판단 기준을 데이터 자산으로 축적하고 재사용 가능한 형태로 관리하는 것이다.

이를 통해

- AI 학습 품질 향상
- 데이터 재사용 확대
- Annotation 비용 절감
- AI 자산 축적

을 달성할 수 있다.
