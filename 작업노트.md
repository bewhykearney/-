# AI-ready Data 가이드 생성 마스터 프롬프트 v1.0

> Part 1 — 목적, 작성 철학, 문체 규칙, 금지 규칙, 용어 규칙

---

# 1. 목적

본 프롬프트의 목적은 AI-ready Data 20개 주제에 대해 계열사가 실제로 구축·운영·확산할 수 있는 수준의 실무 가이드를 일관된 품질로 생성하는 것이다.

가이드는 단순 개념 설명서가 아니다.

가이드는 특정 주제의 필요성, 구성 요소, 구축 방법, 운영 체계, KPI, Roadmap을 포함하는 실행 매뉴얼이다.

문서는 다음 질문에 답해야 한다.

- 이 주제는 AI-ready Data 관점에서 무엇인가
- 왜 필요한가
- 어떤 구성 요소를 갖추어야 하는가
- 어디부터 적용해야 하는가
- 제조업에서는 어떻게 적용되는가
- 데이터 조직과 현업은 무엇을 해야 하는가
- 어떤 산출물과 완료 기준이 필요한가
- 어떻게 운영·확산할 것인가

---

# 2. 최상위 작성 철학

## 2.1 설명보다 실행

가이드는 개념을 설명하기 위해 존재하는 문서가 아니다.

가이드는 독자가 실제로 행동할 수 있도록 만드는 문서이다.

따라서 문서를 작성할 때 항상 아래 질문을 먼저 생각한다.

```text
이 내용을 읽은 사람이
실제로 무엇을 해야 하는가?
```

### 좋지 않은 예

```text
데이터 주석은 데이터에 의미를 부여하는 활동이다.

데이터 주석은 AI 학습에 중요하다.

데이터 주석은 품질이 중요하다.
```

### 좋은 예

```text
데이터 주석 체계를 구축하기 위해서는

1. 대상 데이터를 선정하고
2. Taxonomy를 정의하고
3. Guideline을 작성하고
4. Pilot을 수행한 뒤
5. 본 운영에 착수해야 한다.
```

설명은 행동으로 연결되어야 한다.

독자가 읽은 후 "이해했다" 수준이 아니라 "무엇을 해야 하는지 알겠다" 수준이 되어야 한다.

---

## 2.2 정의는 시작점일 뿐이다

정의는 필요하다.

그러나 정의 자체가 문서의 목적이 되어서는 안 된다.

### 좋지 않은 흐름

```text
정의

정의

정의

정의

예시
```

### 좋은 흐름

```text
문제

왜 필요한가

판단 기준

제조업 사례

수행 방법

산출물

KPI

Roadmap
```

정의는 설명의 시작점이다.

문서는 반드시 실행과 운영 체계까지 연결되어야 한다.

---

## 2.3 AI-ready Data 관점 유지

각 주제는 독립된 기술 주제가 아니다.

모든 주제는 AI-ready Data 체계의 일부로 설명한다.

### 좋지 않은 설명

```text
데이터 전처리는 파서를 만드는 것이다.
```

### 좋은 설명

```text
데이터 전처리는
AI가 데이터를 읽을 수 있도록
구조화하는 활동이다.
```

### 좋지 않은 설명

```text
데이터 해설·주석은
라벨링 작업이다.
```

### 좋은 설명

```text
데이터 해설·주석은
사람의 판단 기준을
데이터 자산으로 전환하는 활동이다.
```

기술 자체보다 AI 활용 관점에서 설명한다.

---

## 2.4 데이터 자산화 관점

모든 주제는 최종적으로 데이터 자산화와 연결되어야 한다.

단발성 프로젝트 결과물이 아니라 재사용 가능한 자산을 만드는 관점으로 작성한다.

문서 작성 시 아래 질문을 항상 고려한다.

```text
이 결과물을
다음 AI 과제에서도
재사용할 수 있는가?
```

가이드는 프로젝트 성공 방법이 아니라 조직 역량 축적 방법을 설명해야 한다.

---

## 2.5 제조업 실무 우선

가이드는 제조업 현업 조직을 대상으로 작성한다.

따라서 일반 소비자 서비스 사례보다 실제 제조업 사례를 우선 사용한다.

### 좋지 않은 사례

```text
스마트폰 사진 분류

고양이 / 강아지 이미지

넷플릭스 추천 시스템

쇼핑몰 상품 추천
```

### 좋은 사례

```text
동판 외관 검사

설비 점검 데이터

고객 클레임

시험 결과 보고서

설비 이상 알람
```

가능하면 두산 계열사에서 실제 활용 가능한 사례를 우선 사용한다.

# 3. 문체 규칙

## 3.1 기본 문체

문체는 컨설팅 보고서 스타일을 따른다.

문장은 짧고 명확하게 작성한다.

그러나 단문만 반복하지 않는다.

### 권장 표현

```text
따라서

반면

이후

결과적으로

특히

예를 들어

이 과정을 통해

다음 단계에서는
```

### 비권장 표현

```text
아무튼

쉽게 말하면

결국은

이런 느낌으로

뚝딱

얼른
```

---

## 3.2 독자 관점

독자는 다음 4개 그룹이다.

| 독자 | 관심사 |
|--------|--------|
| 경영진 | 왜 필요한가, 기대 효과 |
| 현업 SME | 판단 기준, 역할 |
| 데이터 조직 | 구축 방법, 운영 |
| AI 조직 | 활용 방법, 자동화 |

문서는 특정 그룹만 이해할 수 있는 형태로 작성하지 않는다.

---

## 3.3 문장 길이

### 좋지 않은 예

```text
데이터 품질이 중요하다.

데이터 품질은 필요하다.

품질 관리가 중요하다.

품질은 필수다.
```

### 좋은 예

```text
데이터 품질은 AI 성능에 직접적인 영향을 준다.

따라서 데이터 품질 기준을 정의하고,
운영 과정에서 지속적으로 관리해야 한다.
```

문장은 짧게 쓰되, 논리적 연결을 유지한다.

# 4. 금지 규칙

## 4.1 요약 박스 금지

다음 표현은 사용하지 않는다.

```text
한 줄 요약

한 줄 정의

쉽게 말하면

정리하면

핵심은

요약하면

한 문장으로 표현하면

실무 한 줄 정리
```

### 좋지 않은 예

```text
한 줄 요약

데이터 주석은
AI를 위한 정답 데이터이다.
```

### 좋은 예

```text
데이터 주석은
AI 학습에 필요한 의미 정보를
데이터에 부여하는 활동이다.

주석 품질은
AI 성능에 직접적인 영향을 준다.
```

요약 내용은 일반 문단 안으로 흡수한다.

---

## 4.2 이모티콘 금지

다음 표현은 사용하지 않는다.

```text
👉
❓
🏭
🏢
✅
❌
📌
🤖
```

이모티콘 대신 일반 문장을 사용한다.

---

## 4.3 메타 코멘트 금지

다음 표현은 사용하지 않는다.

```text
아래는 작성본입니다

다음은

참고로

제가 작성한 내용은

설명드리자면
```

문서는 바로 본문부터 시작한다.

---

## 4.4 비유 사용 금지

AI는 아래 비유를 매우 자주 생성한다.

```text
스마트폰 사진

고양이

강아지

넷플릭스 추천

쇼핑몰 상품 분류
```

원칙적으로 사용하지 않는다.

실제 제조업 사례를 사용한다.

# 5. 용어 규칙

## 5.1 국문 우선

본문은 국문 중심으로 작성한다.

영문 병기는 최초 정의 시에만 사용한다.

### 좋은 예

```text
데이터 해설·주석(Data Annotation)

현업 전문가(SME)

Human-in-the-Loop(HITL)
```

이후에는

```text
주석

SME

HITL
```

로 통일한다.

---

## 5.2 현업 용어 우선

기술적으로 문제가 없는 경우 현업 용어를 사용한다.

### 지양

```text
CCL

Crack

Defect Class
```

### 권장

```text
동판

균열

결함 유형
```

---

## 5.3 용어 통일

동일 개념은 문서 전체에서 동일 용어를 사용한다.

### 좋지 않은 예

```text
주석

Annotation

라벨링

태깅
```

혼용

### 좋은 예

```text
주석
```

일관 사용

---

## 5.4 제조업 용어 우선 규칙

가능하면 현업에서 실제 사용하는 표현을 우선 사용한다.

### 지양

```text
Copper Clad Laminate

Crack

Delamination
```

### 권장

```text
동판

균열

박리
```

최초 정의 이후에는 대표 용어 하나로 통일한다.

# 6. 문서 구조 규격

모든 AI-ready Data 가이드는 아래 구조를 기본으로 사용한다.

```text
1. 개요

2. 왜 필요한가 (Why)

3. 무엇을 갖추나 (What)

4. 어디부터 하나 (Where)

5. 예시 시나리오

6. 어떻게 준비·운영하나 (How)

7. 관계 또는 KPI 및 성과 관리

8. Roadmap

Appendix
```

주제 특성에 따라 7장은 아래 중 하나를 선택할 수 있다.

```text
다른 주제와의 관계

KPI 및 성과 관리

운영 지표 및 고도화

Governance
```

그러나 나머지 구조는 유지한다.

---

## 6.1 각 장의 역할

### 1장 개요

문서 전체를 한 페이지 안에 설명한다.

설명할 내용

- 주제 정의
- AI-ready Data 체계 내 위치
- 주요 참여 조직
- 문서 범위

설명하지 말아야 할 내용

- 세부 방법론
- KPI
- Roadmap

---

### 2장 Why

왜 필요한지를 설명한다.

설명할 내용

- 현재 문제
- 현업 Pain Point
- 기대 효과

설명하지 말아야 할 내용

- 구축 절차

---

### 3장 What

무엇을 갖추어야 하는지를 설명한다.

설명할 내용

- 정본 모델
- 핵심 구성 요소
- 데이터 구조
- 운영 체계

설명하지 말아야 할 내용

- 상세 실행 절차

---

### 4장 Where

어디부터 적용해야 하는지를 설명한다.

설명할 내용

- 적용 대상
- 우선순위
- 선정 기준

설명하지 말아야 할 내용

- 구축 절차

---

### 5장 Example

실제 사례를 설명한다.

설명할 내용

- End-to-End 수행 흐름
- 실제 제조업 사례

설명하지 말아야 할 내용

- 일반론
- 교과서식 설명

---

### 6장 How

실행 방법론을 설명한다.

가장 중요한 장이다.

설명할 내용

- 실제 수행 절차
- 역할 분담
- 산출물
- 완료 기준

---

### 7장

관계 또는 KPI

주제 특성에 따라 선택한다.

---

### 8장

Roadmap

성숙도 향상 계획을 설명한다.

---

# 7. 정본 모델 규칙

## 7.1 정본 모델의 정의

정본 모델은 해당 주제를 수행하는 핵심 프로세스이다.

모든 주제는 반드시 하나의 정본 모델을 가져야 한다.

---

좋지 않은 예

```text
3장

A → B → C

5장

D → E → F

6장

G → H → I
```

문서마다 다른 흐름

---

좋은 예

```text
3장

A → B → C

5장

A → B → C

6장

A → B → C

8장

A → B → C
```

동일 흐름 사용

---

## 7.2 정본 모델 검증 규칙

문서 생성 후 반드시 확인한다.

```text
3장의 정본 모델이

5장
6장
8장

에서 동일하게 사용되는가
```

---

좋지 않은 예

### 3장

```text
대상 선정
→ 분류 체계
→ 운영
```

### 5장

```text
배경
→ 효과
→ 결과
```

### 6장

```text
수집
→ 분석
→ 활용
```

---

좋은 예

### 3장

```text
대상 선정
→ Taxonomy
→ Guideline
→ Pilot
→ 운영
```

### 5장

```text
대상 선정
→ Taxonomy
→ Guideline
→ Pilot
→ 운영
```

### 6장

```text
대상 선정
→ Taxonomy
→ Guideline
→ Pilot
→ 운영
```

---

## 7.3 단계 수가 다른 경우

단계 수가 달라질 수는 있다.

단 반드시 관계를 설명한다.

예시

### 3장

```text
추출
→ 구조화
→ 위치보존
→ 청킹
```

### 6장

```text
대상 선정
→ 추출
→ 구조화
→ 위치보존
→ 청킹
→ 적재
```

설명

```text
3장은 핵심 변환 단계

6장은 실제 실행 절차
```

# 8. KQ 규칙

## 8.1 KQ의 역할

KQ는 독자를 위한 것이 아니라

문서 설계를 위한 장치이다.

따라서 KQ는 최소한으로 노출한다.

---

## 8.2 KQ 사용 위치

허용

```text
상단 매핑 표
```

---

금지

```text
KQ-1

핵심 질문 2

이 절은 KQ-3에 대한 답변이다
```

---

## 8.3 KQ 매핑 규칙

모든 KQ는 본문에서 답변되어야 한다.

예시

| KQ | 질문 | 위치 |
|------|------|------|
| 1 | 어떤 데이터가 대상인가 | 4장 |
| 2 | 무엇을 갖추어야 하는가 | 3장 |
| 3 | 어떻게 실행하는가 | 6장 |

문서 작성 후 반드시 확인한다.

```text
모든 KQ가 답변되었는가
```

# 9. 5장 예시 시나리오 작성 규칙

## 9.1 5장의 역할

5장은 사례 소개가 아니다.

5장은 정본 모델을 실제 제조업 환경에 적용한 End-to-End 시나리오이다.

---

좋지 않은 예

```text
적용 배경

기대 효과

결과
```

---

좋은 예

```text
대상 선정
→ 기준 설계
→ Pilot
→ 검증
→ 본 운영
→ 자산화
```

---

## 9.2 사례 선택 규칙

하나의 사례를 끝까지 유지한다.

---

권장 사례

```text
동판 외관 검사

설비 점검 데이터

고객 클레임

시험 결과 데이터

설비 이상 탐지
```

---

비권장

```text
5.1

동판

5.2
```

# 10. 6장 How 작성 규칙

## 10.1 6장은 가장 중요한 장이다

독자가 실제로 따라 하는 장은 6장이다.

따라서 6장은 설명서가 아니라 실행 매뉴얼이어야 한다.

---


```text

무엇을 산출하는가

완료 기준은 무엇인가
```

6장을 읽은 독자는


# 11. KPI 작성 규칙

## 11.1 KPI는 품질만 측정하지 않는다

AI-ready Data는 데이터 자산화 체계이다.

따라서 KPI도 자산화 관점을 포함해야 한다.

---

좋지 않은 예

```text
정확도

정밀도

재현율
```

---

좋은 예

```text
품질 KPI

생산성 KPI

운영 KPI

자동화 KPI

자산화 KPI
```

---

## 11.2 품질 KPI

품질 KPI는 데이터 자체의 품질을 측정한다.

예시

| KPI | 설명 |
|--------|--------|
| IAA | 작업자 간 일치도 |
| 오류율 | 잘못된 데이터 비율 |
| QA 통과율 | 검수 통과 비율 |

---

## 11.3 생산성 KPI

생산성 KPI는 구축 효율을 측정한다.

예시

| KPI | 설명 |
|--------|--------|
| 처리량 | 시간당 처리 건수 |
| 건당 비용 | 처리 비용 |
| 리드타임 | 완료까지 걸린 시간 |

---

## 11.4 운영 KPI

운영 KPI는 운영 안정성을 측정한다.

예시

| KPI | 설명 |
|--------|--------|
| 재작업률 | 수정 비율 |
| 운영 중 오류 | 운영 품질 |

---

## 11.5 자동화 KPI

자동화 수준을 측정한다.

예시

| KPI | 설명 |
|--------|--------|
| AI 활용률 | AI 자동 처리 비율 |
| Pre-label 채택률 | AI 주석 승인율 |
| 자동 처리율 | 사람 개입 없는 처리 |

---

## 11.6 자산화 KPI

가장 중요하다.

---

예시

| KPI | 설명 |
|--------|--------|
| Registry 등록률 | 등록된 데이터 비율 |
| 재사용률 | 타 과제 재사용 비율 |
| Gold Dataset 활용률 | 활용 수준 |
| 계열사 적용률 | 확산 수준 |

---
